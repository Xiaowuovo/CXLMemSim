### 7.1 前端整体架构

#### 7.1.1 MVC设计模式应用

**撰写框架**（1.5页）:

```markdown
【行文示例】
前端采用经典的**MVC设计模式**（Model-View-Controller）组织代码结构，实现了业务
逻辑与界面展示的解耦。

**分层职责**

- **Model（模型层）**: 后端TimingAnalyzer、TopologyGraph等核心引擎，负责仿真逻辑
  与数据存储；

- **View（视图层）**: Qt Widget组件（TopologyEditor、MetricsPanel等），负责渲染
  界面与用户交互；

- **Controller（控制器）**: MainWindow类，负责协调Model与View，处理用户操作并
  触发后端计算。

图7-1展示了MVC各层的交互流程：

【图7-1 前端MVC架构交互流程】
```
User Action (e.g., 点击"开始仿真")
      │
      ▼
  MainWindow (Controller)
      │
      ├──> TimingAnalyzer::start()  (Model)
      │         │
      │         ▼ 发射信号
      │    statsUpdated(EpochStats)
      │         │
      └─────────┘
            │
            ▼
      MetricsPanel::updateStats(stats)  (View)
            │
            ▼
      刷新Label、Chart等UI组件
```

**Qt信号槽机制**

Qt的信号槽（Signal-Slot）机制是MVC通信的核心。后端Model通过`emit`发射信号，
前端View的槽函数（Slot）自动响应：

```cpp
// Model发射信号
class TimingAnalyzer : public QObject {
    Q_OBJECT
signals:
    void statsUpdated(const EpochStats& stats);
    void simulationFinished();
};

// Controller连接信号到View
MainWindow::MainWindow() {
    connect(analyzer_, &TimingAnalyzer::statsUpdated,
            metricsPanel_, &MetricsPanel::updateStats);
    
    connect(analyzer_, &TimingAnalyzer::simulationFinished,
            this, &MainWindow::onSimulationFinished);
}
```

这种**观察者模式**（Observer Pattern）确保了Model状态变化时，所有订阅的View
自动同步更新，无需显式调用刷新函数，降低了耦合度。
```

- 代码示例展示Qt信号槽的实际应用

#### 7.2.1 GraphicsView场景管理

**撰写框架**（2页）:

```markdown
【行文示例】
拓扑编辑器基于Qt的**GraphicsView框架**实现，该框架采用场景-视图（Scene-View）

**绘制实现**

```cpp
void TopologyNodeItem::paint(QPainter* painter, const QStyleOptionGraphicsItem*, QWidget*) {
    painter->setRenderHint(QPainter::Antialiasing);
    
    // 绘制节点矩形
    QRectF rect = boundingRect();
    painter->setBrush(color_);
    painter->setPen(QPen(Qt::white, 2));
    painter->drawRoundedRect(rect, 8, 8);
    
    // 绘制节点ID文本
    painter->setPen(Qt::white);
    painter->setFont(QFont("Arial", 10, QFont::Bold));
    painter->drawText(rect, Qt::AlignCenter, id_);
    
    // 绘制连接锚点（四个小圆圈）
    painter->setBrush(Qt::white);
    painter->drawEllipse(rect.topLeft() + QPointF(rect.width()/2, 0), 4, 4);     // 上
    painter->drawEllipse(rect.bottomLeft() + QPointF(rect.width()/2, 0), 4, 4);  // 下
    painter->drawEllipse(rect.topLeft() + QPointF(0, rect.height()/2), 4, 4);    // 左
    painter->drawEllipse(rect.topRight() + QPointF(0, rect.height()/2), 4, 4);   // 右
}

void TopologyNodeItem::mouseMoveEvent(QGraphicsSceneMouseEvent* event) {

#### 7.2.2 连线路径绘制算法

**撰写框架**（1.5页）:

```markdown
【行文示例】
节点间的连接线需要智能避让节点，并在两端绘制箭头。本系统采用**贝塞尔曲线**
（Bezier Curve）绘制平滑的连接路径。

**路径计算**

给定源节点锚点P0与目标节点锚点P3，通过两个控制点P1、P2生成三次贝塞尔曲线：

```cpp
QPainterPath TopologyEdgeItem::calculate_path() const {
    QPointF p0 = src_anchor_pos_;  // 源锚点
    QPointF p3 = dst_anchor_pos_;  // 目标锚点
    
    // 计算控制点（沿连线方向偏移）
    QPointF offset = (p3 - p0) * 0.3;
    QPointF p1 = p0 + offset;
    QPointF p2 = p3 - offset;
    
    // 创建贝塞尔曲线路径
    QPainterPath path;
    path.moveTo(p0);
    path.cubicTo(p1, p2, p3);
    
    return path;
}
```

**箭头绘制**

在路径终点绘制三角形箭头，方向沿切线方向：

```cpp
void TopologyEdgeItem::paint(QPainter* painter, const QStyleOptionGraphicsItem*, QWidget*) {
    // 绘制曲线
    painter->setPen(QPen(Qt::gray, 2));
    painter->drawPath(path_);
    
    // 计算箭头位置与方向
    qreal t = 0.95;  // 在95%位置绘制箭头
    QPointF arrow_pos = path_.pointAtPercent(t);
    qreal angle = path_.angleAtPercent(t);
    
    // 绘制三角形箭头
    QPolygonF arrow_head;
    arrow_head << QPointF(0, 0)
               << QPointF(-10, 5)
               << QPointF(-10, -5);
    
    painter->save();
    painter->translate(arrow_pos);
    painter->rotate(-angle);
    painter->setBrush(Qt::gray);
    painter->drawPolygon(arrow_head);
    painter->restore();
}
```

void RealTimeChartWidget::appendDataPoint(uint64_t epoch_id, double avg_latency, double p95_latency) {
    avg_latency_series_->append(epoch_id, avg_latency);
    p95_latency_series_->append(epoch_id, p95_latency);
    
    // 移除超出窗口的旧数据
    if (avg_latency_series_->count() > 100) {
        avg_latency_series_->remove(0);
        p95_latency_series_->remove(0);
    }
    
    // 动态调整X轴范围
    auto* axisX = qobject_cast<QValueAxis*>(chart_->axes(Qt::Horizontal).first());
    if (epoch_id > 100) {
        axisX->setRange(epoch_id - 100, epoch_id);
    }
    
    // 动态调整Y轴范围（留10%余量）
    double max_val = std::max(avg_latency_series_->at(avg_latency_series_->count()-1).y(),
                              p95_latency_series_->at(p95_latency_series_->count()-1).y());
    auto* axisY = qobject_cast<QValueAxis*>(chart_->axes(Qt::Vertical).first());
    axisY->setRange(0, max_val * 1.1);
}

void MainWindow::onExportConfigClicked() {

#### 8.1.2 基准负载选择

**撰写框架**（1.5页）:

```markdown
【行文示例】
实验选取了三类具有代表性的基准负载（Benchmark）：

**（1）STREAM内存带宽测试**

STREAM是业界标准的内存带宽基准测试，包含Copy、Scale、Add、Triad四种访问模式[1]。
本实验使用Triad模式（`a[i] = b[i] + c[i] * scalar`），数组大小1GB，迭代100次。

- **访问特征**: 顺序流式访问（Sequential Streaming），L3 Miss率>90%
- **仿真目的**: 评估CXL设备在高带宽场景下的性能上限

**（2）SPEC CPU2017 mcf**

mcf是SPEC CPU2017整数基准套件中的内存密集型负载，实现单纯形法求解最小费用流
问题[2]。

- **访问特征**: 随机访问（Random Access），访问地址跨度大，L3 Miss率60-70%
- **工作集**: 约2GB（超出L3缓存容量）
- **仿真目的**: 评估CXL设备在随机访问模式下的延迟表现

**（3）YOLOv8目标检测Trace**

使用perf mem录制YOLOv8推理过程的内存访问轨迹（详见4.4.3节），回放该Trace进行
仿真。

- **访问特征**: 混合模式（顺序+随机），呈现周期性突发
- **Trace时长**: 5秒（约120万次访问）
- **仿真目的**: 验证系统对真实AI负载的建模能力

表8-2总结了三种负载的特性对比：

【表8-2 基准负载特性对比】
| 负载 | 访问模式 | L3 Miss率 | 工作集大小 | 适用场景 |
|-----|---------|----------|----------|---------|
| STREAM | 顺序流式 | >90% | 1GB | 带宽评估 |
| mcf | 随机 | 60-70% | 2GB | 延迟评估 |
| YOLOv8 | 混合突发 | 30-40% | 1.5GB | 真实负载 |
```

**话术要点**:
- 引用业界标准基准增强权威性
- 详细描述访问特征便于理解
- 表格对比三种负载的差异
- 说明选择每种负载的目的

### 8.2 拓扑配置对性能的影响

#### 8.2.1 实验设计

**撰写框架**（1页）:

```markdown
【行文示例】
为研究拓扑结构对CXL访问延迟的影响，设计了以下四种拓扑配置作为对照组：

【图8-1 四种实验拓扑配置】
```
Topo-1 (Direct)        Topo-2 (1-Switch)
   RC0                    RC0
    │                      │
   DEV0                   SW1
                          / \
                      DEV0 DEV1

Topo-3 (2-Switch)      Topo-4 (3-Layer)
   RC0                    RC0
    │                      │
   SW1                    SW1
   / \                    / \
 SW2 SW3              SW2   SW3
  │   │               / \   / \
DEV0 DEV1         DEV0 DEV1 DEV2 DEV3
```

**控制变量**:
- 所有链路均为PCIe 5.0 x16（64GB/s）
- 所有设备基础延迟均为170ns（DDR5-4800）
- 工作负载固定为STREAM Triad

**实验步骤**:
1. 在每种拓扑下运行STREAM基准测试
2. 记录10个Epoch的平均延迟与P95延迟
3. 重复运行5次取平均值
```

#### 8.2.2 实验结果与分析

**撰写框架**（2页）:

```markdown
【行文示例】
图8-2展示了四种拓扑下的平均延迟对比（误差线表示标准差）：

【图8-2 拓扑配置对平均延迟的影响】
（柱状图：横轴为Topo-1至Topo-4，纵轴为延迟/ns，柱子高度递增，带误差线）

【表8-3 拓扑延迟测量结果】
| 拓扑 | 平均延迟(ns) | P95延迟(ns) | 相对Topo-1增长 |
|-----|------------|-----------|--------------|
| Topo-1 | 185.2 ± 3.1 | 212.5 | - (基准) |
| Topo-2 | 207.8 ± 4.2 | 241.3 | +12.2% |
| Topo-3 | 229.1 ± 5.8 | 278.6 | +23.7% |
| Topo-4 | 231.5 ± 6.1 | 283.2 | +25.0% |

**结果分析**

（1）**链路跳数主导延迟**: Topo-1（直连）延迟最低，每增加一层Switch，延迟增加
    约20-25ns（对应一条链路的双向传播延迟）；

（2）**Topo-3与Topo-4延迟接近**: 尽管Topo-4有4个设备而Topo-3只有2个，但由于
    STREAM的访问是顺序的，不会触发多设备并发，因此设备数量对延迟影响不大；

（3）**P95延迟增长更显著**: Topo-4的P95延迟比Topo-1高33.3%，表明复杂拓扑下的
    尾延迟问题更严重（拥塞导致的排队延迟）。

**延迟分解分析**

图8-3展示了Topo-1与Topo-4的延迟Breakdown对比：

【图8-3 Topo-1 vs Topo-4 延迟分解对比】
（双柱堆叠图：清晰显示设备延迟占比下降，链路延迟占比上升）

可见，Topo-4中链路延迟占比从Topo-1的8%上升至24%，成为延迟增长的主要来源。

**优化建议**

对于延迟敏感型应用（如实时数据库），应优先选择扁平化拓扑（减少Switch层数），
牺牲可扩展性换取低延迟。
```

**话术要点**:
- 用均值±标准差格式增强数据可信度
- 逐条分析实验发现并解释原因
- 用Breakdown图定位瓶颈
- 给出实际的优化建议

### 8.3 地址映射策略对比实验

#### 8.3.1 实验设计

**撰写框架**（0.5页）:

```markdown
【行文示例】
为对比First-Touch、Interleave、Tiering三种映射策略的性能差异，固定拓扑为Topo-2
（1个Switch + 2个Device），分别测试三种策略在mcf负载下的表现。

**参数配置**:
- Tiering热度阈值: 200ns
- Tiering迁移周期: 10个Epoch
- 每种策略运行100个Epoch（约1秒）
```

#### 8.3.2 实验结果

**撰写框架**（2页）:

```markdown
【行文示例】
图8-4展示了三种策略下的平均延迟随时间的变化曲线：

【图8-4 映射策略延迟时序对比】
（折线图：横轴为Epoch，纵轴为延迟，三条曲线：First-Touch波动大，Interleave平稳但高，
Tiering开始高后迅速下降并趋于稳定）

【表8-4 映射策略性能对比】
| 策略 | 平均延迟 | P95延迟 | 设备利用率均衡度 | 页面迁移次数 |
|-----|---------|--------|---------------|-----------|
| First-Touch | 245.3 ns | 412.7 ns | 0.34 (不均衡) | 0 |
| Interleave | 238.1 ns | 385.2 ns | 0.92 (均衡) | 0 |
| Tiering | **218.6 ns** | **351.8 ns** | 0.87 (较均衡) | 1247 |

*设备利用率均衡度 = 1 - (利用率标准差 / 平均利用率)*

**结果分析**

（1）**Interleave优于First-Touch**: Interleave通过轮询分配实现了设备负载均衡，
    避免了First-Touch的热点集中问题，平均延迟降低2.9%；

（2）**Tiering最优**: Tiering策略通过动态迁移热页至DRAM（本实验中模拟为低延迟
    CXL设备），平均延迟比Interleave进一步降低8.2%，P95延迟降低8.7%；

（3）**迁移开销可控**: Tiering产生了1247次页面迁移（平均每Epoch 12.5次），但由于
    迁移发生在Epoch结束时（仿真暂停期间），未对仿真性能造成影响。实际部署中需
    评估迁移开销对应用吞吐量的影响。

图8-5展示了Tiering策略下热页命中率随时间的提升：

【图8-5 Tiering热页命中率提升曲线】
（折线图：横轴为Epoch，纵轴为DRAM命中率，曲线从40%逐步上升至75%并稳定）

可见，经过约20个Epoch的学习期，Tiering策略成功将75%的热页迁移至DRAM，显著
降低了平均访问延迟。

**适用场景建议**

- **First-Touch**: 访问模式高度局部化，且用户明确知道数据分布的场景
- **Interleave**: 访问模式随机均匀，需要最大化并发带宽的场景（如图数据库）
- **Tiering**: 访问模式存在明显冷热分层，且能容忍短暂学习期的场景（推荐）
```

### 8.4 真实负载案例研究

#### 8.4.1 YOLOv8推理性能优化

**撰写框架**（2页）:

```markdown
【行文示例】
为验证系统对真实AI负载的指导价值，本节以YOLOv8目标检测模型为例，探索CXL内存
配置对推理性能的影响。

**实验场景**

某边缘AI服务器部署YOLOv8模型进行实时视频分析，受成本限制，DRAM容量仅64GB，
但模型权重+激活值+视频缓冲区需约80GB内存。考虑引入CXL扩展内存来避免频繁换页。

**对比配置**

- **Baseline**: 纯DRAM（64GB），超出部分使用Swap到SSD（延迟约100μs）
- **Config-A**: DRAM (64GB) + CXL Device (16GB, Direct连接)
- **Config-B**: DRAM (64GB) + CXL Device (32GB, 经Switch连接)

**实验结果**

【表8-5 YOLOv8推理性能对比】
| 配置 | 帧率(FPS) | 平均帧延迟(ms) | P99帧延迟(ms) | 内存命中率 |
|-----|----------|--------------|-------------|----------|
| Baseline | 18.3 | 54.6 | 127.8 | 78% (频繁Swap) |
| Config-A | **28.7** | **34.9** | **51.2** | 95% |
| Config-B | 26.1 | 38.3 | 58.7 | 94% |

**性能提升显著**

相比Baseline，Config-A将帧率提升**56.8%**，P99延迟降低**60.0%**，几乎消除了
Swap导致的长尾延迟。Config-B由于多一跳Switch，性能略逊于Config-A，但仍优于
Baseline。

图8-6展示了三种配置下的帧延迟CDF（Cumulative Distribution Function）曲线：

【图8-6 帧延迟CDF曲线对比】
（横轴：延迟/ms，纵轴：累积概率0-1，三条S型曲线，Config-A最左（最优））

Config-A的99%分位点在51.2ms，而Baseline的99%分位点高达127.8ms，证明CXL扩展
内存有效消除了SSD Swap的长尾延迟。

**经济性分析**

假设：
- DDR5-4800 64GB ≈ $300
- CXL扩展内存16GB ≈ $80（预计2025年价格[3]）
- PCIe 5.0 SSD 1TB ≈ $150

Config-A总成本 = $300 + $80 = $380，仅比Baseline高26.7%，但性能提升56.8%，
性价比显著。

**结论与建议**

对于内存容量受限的边缘AI服务器，采用DRAM + 小容量CXL扩展内存的混合配置是
一种低成本高性能方案。建议将模型权重（冷数据）放置于CXL设备，激活值（热数据）
保留在DRAM，可实现接近纯DRAM的性能表现。
```

**话术要点**:
- 结合实际应用场景增强实用价值
- 给出详细的性能提升数据与百分比
- 用CDF曲线直观展示尾延迟差异
- 提供经济性分析体现产品化思维
- 给出明确的优化建议

**撰写框架**（1页）:
