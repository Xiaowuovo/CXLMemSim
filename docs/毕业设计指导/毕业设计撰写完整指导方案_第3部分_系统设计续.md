**自动布局算法**

为避免节点重叠与链路交叉，系统实现了基于**力导向图**（Force-Directed Graph）的
自动布局算法：

```cpp
void TopologyEditor::auto_layout() {
    const double k = 100.0;  // 弹簧劲度系数
    const int iterations = 100;
    
    for (int iter = 0; iter < iterations; ++iter) {
        // 计算斥力（所有节点对之间）
        for (auto& node1 : nodes_) {
            QPointF force(0, 0);
            for (auto& node2 : nodes_) {
                if (node1 == node2) continue;
                QPointF delta = node1->pos() - node2->pos();
                double dist = std::max(delta.manhattanLength(), 1.0);
                force += delta / dist * k * k / dist;  // 库仑斥力
            }
            
            // 计算引力（连接的节点对之间）
            for (auto& edge : node1->edges()) {
                QPointF delta = edge->target()->pos() - node1->pos();
                double dist = delta.manhattanLength();
                force += delta * dist / k;  // 胡克引力
            }
            
            // 更新位置
            node1->setPos(node1->pos() + force * 0.01);  // 阻尼系数0.01
        }
    }
}
```

该算法使节点间距自然分布，链路长度趋于均匀，提升可视化效果。

### 5.3 最短路径发现算法

#### 5.3.1 Dijkstra算法应用

**撰写框架**（2页）:

```markdown
【行文示例】
给定源节点（Root Complex）与目标节点（CXL Device），系统需要计算拓扑图中的最短
延迟路径。本系统采用经典的**Dijkstra算法**实现最短路径发现，并针对CXL拓扑特点
进行优化。

**算法原理**

Dijkstra算法维护一个最小优先队列（Min-Heap），每次选择当前距离源节点最近的未
访问节点进行松弛（Relaxation）操作。算法复杂度为O((V+E) log V)，其中V为节点数，
E为边数。

对于CXL拓扑场景，V通常在10-100量级（1个RC + 若干Switch + 若干Device），E在
20-200量级，算法性能可忽略不计（微秒级）。

**实现代码**

```cpp
TopologyPath TopologyGraph::find_shortest_path(
    const std::string& src_id,
    const std::string& dst_id
) const {
    // 初始化距离表
    std::unordered_map<std::string, double> dist;
    std::unordered_map<std::string, std::string> prev;  // 前驱节点
    for (const auto& [id, node] : nodes_) {
        dist[id] = std::numeric_limits<double>::infinity();
    }
    dist[src_id] = 0.0;
    
    // 最小优先队列（距离，节点ID）
    using QueueElem = std::pair<double, std::string>;
    std::priority_queue<QueueElem, std::vector<QueueElem>, std::greater<>> pq;
    pq.push({0.0, src_id});
    
    while (!pq.empty()) {
        auto [curr_dist, curr_id] = pq.top();
        pq.pop();
        
        if (curr_id == dst_id) break;  // 提前终止优化
        if (curr_dist > dist[curr_id]) continue;  // 过时条目
        
        // 松弛操作
        for (const auto& edge_id : nodes_.at(curr_id).connected_edges) {
            const auto& edge = edges_.at(edge_id);
            if (edge.src_node_id != curr_id) continue;  // 只考虑出边
            
            std::string neighbor_id = edge.dst_node_id;
            double new_dist = curr_dist + edge.spec.latency_ns;
            
            if (new_dist < dist[neighbor_id]) {
                dist[neighbor_id] = new_dist;
                prev[neighbor_id] = curr_id;
                pq.push({new_dist, neighbor_id});
            }
        }
    }
    
    // 回溯路径
    TopologyPath path;
    std::string curr = dst_id;
    while (curr != src_id) {
        path.nodes.push_front(curr);
        curr = prev[curr];
    }
    path.nodes.push_front(src_id);
    path.total_latency_ns = dist[dst_id];
    
    return path;
}
```

**CXL特定优化**

（1）**提前终止**：一旦到达目标设备，立即停止搜索（无需遍历全图）；

（2）**单源多目标缓存**：RC到各设备的路径在初始化时预计算并缓存，运行时O(1)查询；

（3）**路径有效性检测**：若路径上任一链路利用率超过90%，标记为"拥塞路径"，触发
负载均衡重路由。

#### 5.3.2 多路径负载均衡

**撰写框架**（1.5页）:

```markdown
【行文示例】
在多Switch拓扑中，RC到某个Device可能存在多条等长路径。为避免单条路径过载，系统
实现了基于**加权轮询**（Weighted Round-Robin）的多路径负载均衡。

**路径权重计算**

每条路径的权重由路径上所有链路的剩余带宽决定：

```
Path_Weight = min(Link_i.remaining_bandwidth) for all Link_i in Path
```

剩余带宽计算：
```cpp
double Link::get_remaining_bandwidth() const {
    double total_bw = spec.get_bandwidth_gbps() * 1e9;  // bps
    double used_bw = bytes_transferred / epoch_duration_s * 8;
    return total_bw - used_bw;
}
```

**加权轮询调度**

```cpp
std::string TopologyGraph::select_path_for_access(
    const std::string& device_id,
    const std::vector<TopologyPath>& candidate_paths
) {
    // 计算总权重
    double total_weight = 0.0;
    for (const auto& path : candidate_paths) {
        total_weight += path.get_weight();
    }
    
    // 加权随机选择
    double rand_val = uniform_random(0, total_weight);
    double cumulative = 0.0;
    for (const auto& path : candidate_paths) {
        cumulative += path.get_weight();
        if (rand_val <= cumulative) {
            return path.id;
        }
    }
    
    return candidate_paths[0].id;  // 兜底
}
```

### 5.4 地址映射策略

#### 5.4.1 First-Touch策略

**撰写框架**（1页）:

```markdown
【行文示例】
First-Touch策略遵循操作系统的NUMA默认行为：首次访问某页面的CPU核心所在的NUMA
节点获得该页面的所有权。在CXL场景中，若首次访问来自本地DRAM，则该页面保留在
DRAM；若首次访问触发Page Fault且DRAM已满，则分配至CXL设备。

**优点**：
- 自然反映应用的内存访问局部性
- 无需预先了解访问模式

**缺点**：
- 可能导致CXL设备利用率不均（热点集中在某个设备）
- 首次访问延迟较高（Page Fault开销）

**实现机制**

```cpp
std::string MemoryPolicy::allocate_first_touch(uint64_t vaddr, uint32_t cpu_id) {
    // 查询CPU所在NUMA节点
    uint32_t numa_node = get_numa_node_of_cpu(cpu_id);
    
    // 优先尝试本地DRAM
    if (dram_nodes_[numa_node].has_free_space()) {
        return dram_nodes_[numa_node].allocate_page(vaddr);
    }
    
    // DRAM已满，分配至CXL设备
    std::string device_id = select_least_loaded_device();
    return devices_[device_id].allocate_page(vaddr);
}
```
```

#### 5.4.3 Tiering分层策略

**撰写框架**（1.5页）:

```markdown
【行文示例】
Tiering策略将内存分为"热层"（DRAM）与"冷层"（CXL），根据访问频率动态迁移页面。
这是一种**软件定义的分层存储**（Software-Defined Memory Tiering）方案。

**热度跟踪机制**

系统为每个页面维护一个访问计数器（Access Counter），Epoch结束时根据计数器值
排序页面：

```cpp
struct PageHeatInfo {
    uint64_t page_addr;
    uint32_t access_count;      // 本Epoch内的访问次数
    double avg_latency_ns;      // 平均访问延迟
    std::string current_tier;   // "DRAM" | "CXL"
};
```

**迁移策略**

每隔N个Epoch（如10个Epoch = 100ms），执行一次页面迁移：

1. 计算热度阈值：`threshold = total_accesses / total_pages * 0.1`（前10%为热页）
2. 选择热页（access_count > threshold）迁移至DRAM
3. 选择冷页（access_count < threshold / 10）迁移至CXL
4. 限制单次迁移量（如最多迁移1GB），避免迁移风暴

```cpp
void MemoryPolicy::perform_tiering_migration() {
    // 按访问计数排序
    std::sort(pages_.begin(), pages_.end(), [](const auto& a, const auto& b) {
        return a.access_count > b.access_count;
    });
    
    size_t num_hot_pages = pages_.size() * 0.1;  // 前10%
    uint64_t migrated_bytes = 0;
    const uint64_t max_migration = 1ULL << 30;  // 1GB
    
    for (size_t i = 0; i < num_hot_pages && migrated_bytes < max_migration; ++i) {
        if (pages_[i].current_tier == "CXL") {
            migrate_page(pages_[i].page_addr, "DRAM");
            migrated_bytes += PAGE_SIZE;
        }
    }
    
    // 迁移冷页至CXL
    for (size_t i = pages_.size() - 1; i > num_hot_pages && migrated_bytes < max_migration; --i) {
        if (pages_[i].current_tier == "DRAM") {
            migrate_page(pages_[i].page_addr, "CXL");
            migrated_bytes += PAGE_SIZE;
        }
    }
}
```

**迁移开销建模**

页面迁移需要拷贝数据，引入额外延迟：

```
Migration_Latency = Page_Size / Min_Bandwidth + TLB_Shootdown_Cost
                  = 4KB / 64GB/s + 5μs
                  ≈ 5.06 μs
```

系统在统计中记录迁移次数与总开销，用于评估Tiering策略的收益。

#### 6.1.3 链路延迟累加

**撰写框架**（1页）:

```markdown
【行文示例】
链路延迟包括**传播延迟**（Propagation Delay）与**传输延迟**（Transmission Delay）。

**传播延迟**

Propagation_Delay = Cable_Length / (3×10^8 m/s × 0.6)
```

对于数据中心内短距离（<1m）连接，传播延迟可忽略（<5ns）。配置中的`latency_ns`
字段主要反映PHY层（物理层）的串化/解串化（SerDes）延迟，典型值10-30ns。

**传输延迟**

单个Flit（68字节，含64字节数据+4字节CRC）的传输时间：
```
Transmission_Delay = Flit_Size / Link_Bandwidth
                   = 68 Bytes / 64 GB/s
                   = 1.06 ns
```

该延迟已包含在设备的基础延迟中（控制器发送至接收Flit的时间），无需重复计算。

**路径延迟累加实现**

```cpp
double LatencyModel::calculate_path_latency(const TopologyPath& path) {
    double total_latency = 0.0;
    for (const auto& edge_id : path.edges) {
        const auto& edge = topology_->get_edge(edge_id);
        total_latency += edge.spec.latency_ns;
    }
    return total_latency;
}
```

#### 6.1.4 协议开销建模

**撰写框架**（1.5页）:

```markdown
【行文示例】
CXL.mem协议基于PCIe 5.0物理层，采用Flit模式传输。每个内存读请求需要经历以下步骤：

1. **请求Flit发送**（RC → Device）：包含地址、命令类型、事务ID
2. **设备处理**：地址解码、DRAM访问
3. **响应Flit发送**（Device → RC）：包含数据、完成状态

**Flit编解码开销**

每个Flit需要附加4字节CRC校验，并进行128b/130b编码（PCIe 5.0）。编解码由硬件
PHY自动完成,延迟约5-10ns。

**重试机制开销**

若CRC校验失败或接收方缓冲区满，发送方需重传Flit。假设误码率（BER）为10^-12，
64字节数据包的重传概率：

```
Retransmission_Prob = 1 - (1 - BER)^(64×8) ≈ 5.12 × 10^-10
```

重传概率极低，仿真中可忽略。但系统仍支持配置重传惩罚：

```json
{
  "protocol": {
    "cxl_version": "2.0",
    "retransmission_penalty_ns": 50,
    "retransmission_rate": 0.0001
  }
}
```

**TLP/DLLP封装开销**

CXL.mem复用PCIe的TLP（Transaction Layer Packet）格式，需附加20字节头部（含地址、
长度、类型字段）。头部传输时间已计入Flit传输延迟，无需单独建模。

综合以上因素，协议开销设定为固定值30ns（可在配置文件中调整）。

### 6.2 拥塞建模

#### 6.2.1 链路利用率与排队延迟

**撰写框架**（2页）:

```markdown
【行文示例】
当链路利用率接近100%时，新到达的访问请求需要在队列中等待，引入额外的排队延迟。
本系统采用**M/M/1排队模型**估算平均排队延迟。

**M/M/1模型假设**

- 请求到达过程服从泊松分布（Poisson Process），到达率为λ
- 服务时间（传输Flit时间）服从指数分布，服务率为μ
- 单个队列，先到先服务（FCFS）

**平均排队延迟公式**

```
Average_Queue_Latency = 1 / (μ - λ)
```

其中，μ = Link_Bandwidth / Flit_Size（单位：Flit/s）

利用率ρ = λ / μ，则：

```
Average_Queue_Latency = (Service_Time) / (1 - ρ)
```

当ρ → 1时，延迟趋于无穷大（链路饱和）。

**数值示例**

假设64 GB/s链路，Flit大小68字节：
```
μ = 64 GB/s / 68 B = 941,176,470 Flit/s
Service_Time = 1 / μ = 1.06 ns
```

不同利用率下的排队延迟：

【表6-1 链路利用率与排队延迟关系】
| 利用率ρ | 平均排队延迟 | 总延迟增加 |
|--------|------------|----------|
| 50%    | 2.12 ns    | +2 ns    |
| 70%    | 3.53 ns    | +3 ns    |
| 80%    | 5.30 ns    | +5 ns    |
| 90%    | 10.6 ns    | +11 ns   |
| 95%    | 21.2 ns    | +21 ns   |
| 99%    | 106 ns     | +106 ns  |

可见，利用率超过90%后，排队延迟急剧增加，成为主要瓶颈。

**实现代码**

```cpp
double LatencyModel::calculate_congestion_penalty(const TopologyEdge& edge) {
    double utilization = edge.current_utilization;
    if (utilization >= 0.99) {
        utilization = 0.99;  // 防止除零
    }
    
    double service_time_ns = (68.0 * 8) / (edge.spec.get_bandwidth_gbps() * 1e9) * 1e9;
    double queue_latency_ns = service_time_ns / (1 - utilization);
    
    return queue_latency_ns;
}
```

图6-2展示了排队延迟随利用率的非线性增长曲线：

【图6-2 链路利用率vs排队延迟曲线】
（横轴：利用率0-100%，纵轴：排队延迟/ns，曲线呈指数增长）
```

**话术要点**:
- 引用排队论经典模型增强理论深度
- 公式推导 + 数值示例增强可信度
- 表格量化不同利用率的影响

### 6.3 MLP并行优化

#### 6.3.1 Memory-Level Parallelism原理

**撰写框架**（1.5页）:

```markdown
【行文示例】
现代处理器支持同时发起多个内存访问请求（Outstanding Misses），这些并发请求可以
共享链路传输时间，从而降低总延迟。这种现象称为**内存级并行**（Memory-Level 
Parallelism, MLP）。

**示例场景**

假设某Epoch内捕获到4次L3 Miss，均访问同一CXL设备，到达时间间隔<10ns（认为并发）。
若不考虑MLP，总延迟为：
```
Total_Latency_Sequential = 4 × 260ns = 1040ns
```

但实际上，4个请求的响应Flit可以流水线（Pipeline）传输，总延迟仅为：
```
Total_Latency_Pipelined = 260ns + 3 × Flit_Transmission_Time
                        = 260ns + 3 × 1.06ns
                        ≈ 263ns
```

MLP加速比 = 1040 / 263 ≈ 3.95倍

**MLP窗口检测**

系统维护一个时间窗口（默认50ns），窗口内到达的访问视为并发：

```cpp
struct MLPWindow {
    uint64_t start_time_ns;
    std::vector<MemoryAccessEvent> concurrent_accesses;
    
    bool can_merge(uint64_t new_time_ns) const {
        return (new_time_ns - start_time_ns) < 50;
    }
};
```

**MLP收益计算**

```cpp
double LatencyModel::calculate_mlp_speedup(const std::vector<MemoryAccessEvent>& accesses) {
    if (accesses.size() <= 1) return 0.0;
    
    double sequential_latency = accesses.size() * base_latency_;
    double pipelined_latency = base_latency_ + (accesses.size() - 1) * flit_time_;
    
    return sequential_latency - pipelined_latency;
}
```

实验表明，MLP可将多访问场景的总延迟降低30-70%（取决于并发度）。
```

LatencyBreakdown TimingAnalyzer::compute_avg_breakdown(const std::vector<AccessRecord>& records) {
    LatencyBreakdown breakdown = {};
    
    for (const auto& record : records) {
        breakdown.device_latency_ns += record.device_latency;
        breakdown.link_latency_ns += record.link_latency;
        breakdown.protocol_overhead_ns += record.protocol_overhead;
        breakdown.congestion_penalty_ns += record.congestion_penalty;
    }
    
    size_t count = records.size();
    breakdown.device_latency_ns /= count;
    breakdown.link_latency_ns /= count;
    breakdown.protocol_overhead_ns /= count;
    breakdown.congestion_penalty_ns /= count;
    
    return breakdown;
}
