# 第4章 子系统设计与实现

## 4.1 底层性能采样与追踪子系统

### 4.1.1 ITracer抽象接口设计

追踪子系统是整个仿真平台的数据入口，其设计质量直接决定了系统的适用范围。本课题面临一个实际矛盾：在物理机上Intel PEBS能提供最精确的L3 Miss采样，但在开发者常用的虚拟机中PEBS完全不可用。解决这一矛盾的关键是设计充分解耦的抽象层，让上层逻辑完全不感知底层采样方式的差异。

为此，本系统定义了ITracer纯虚基类，作为所有追踪器实现的统一契约，接口方法围绕"生命周期管理"和"数据采集"两条主线设计：

```cpp
class ITracer {
public:
    virtual ~ITracer() = default;
    virtual bool initialize(pid_t target_pid) = 0;
    virtual bool start() = 0;
    virtual void stop() = 0;
    virtual std::vector<MemoryAccessEvent> collect_samples() = 0;
    virtual std::string name() const = 0;
    virtual bool supports_precise_address() const = 0;
};
```

`collect_samples()`是与TimingAnalyzer交互的唯一数据出口，每个Epoch调用一次，返回本时间片内捕获的所有`MemoryAccessEvent`。这种批量拉取（Pull）而非事件推送（Push）的设计，避免了高频采样下的信号槽调用开销，也让Epoch边界的统计聚合更为自然。

`MemoryAccessEvent`结构体封装了单次访问的完整上下文：

```cpp
struct MemoryAccessEvent {
    uint64_t timestamp_ns;
    uint64_t virtual_addr;
    uint64_t physical_addr;  // 仅PEBS有效
    uint32_t tid;
    uint32_t cpu;
    uint64_t latency_cycles;
    bool is_load;
    bool is_l3_miss;
    uint32_t data_source;    // PEBS原始data_src字段
};
```

`physical_addr`字段对地址到CXL设备的映射至关重要，因为内存分配策略操作的是物理地址空间。PEBSTracer直接从PEBS记录中获取此字段，MockTracer则通过伪随机方式模拟物理地址分布。

TracerFactory负责在运行时根据环境选择最优Tracer实例，按优先级依次检测：首先检查`/proc/sys/kernel/perf_event_paranoid`是否允许用户态访问PMU（值≤1则支持PEBS），其次检查是否指定了Trace文件路径，两者皆不满足时默认使用MockTracer。这一自动检测机制使同一份可执行程序在不同环境下自动选择最优路径，无需用户手动干预。

**表4-1 三种Tracer能力对比**

| 类型 | 运行环境 | 地址精度 | 仿真开销 | 适用场景 |
|-----|--------|--------|--------|--------|
| PEBSTracer | 物理Intel机 | 精确 | 5-10% | 生产环境验证 |
| MockTracer | 任意环境 | 模拟 | <1% | 虚拟机/原型开发 |
| TraceDriven | 任意环境 | 回放 | ~0% | 可复现实验 |

### 4.1.2 MockTracer合成流量实现

MockTracer的核心挑战是在不执行真实负载的情况下生成统计规律具有代表性的访问序列。真实应用的内存访问呈现两个显著特征：地址空间上的热点集中性和时间维度上的局部性。MockTracer通过两个数学模型分别刻画这两个特征。

**访问地址建模**：系统将地址空间分为Hot区域（占总空间10%，承载70%访问）和Cold区域，Hot区域内按Zipf分布（偏斜参数α=0.9）采样，Cold区域按均匀分布采样：

```cpp
uint64_t MockTracer::generate_address() {
    double r = uniform_random(0, 1);
    if (r < 0.7) {
        uint32_t rank = zipf_sample(num_hot_pages_, 0.9);
        return hot_pages_[rank];
    }
    return uniform_sample(cold_region_start_, cold_region_end_);
}
```

**L3 Miss率模拟**：系统内置容量可配置的LRU缓存模拟LLC行为，访问命中缓存则标记`is_l3_miss=false`，未命中则标记为true并淘汰最久未用条目。LRU容量与Miss率的对应关系通过预实验标定写入配置，用户直接指定目标Miss率即可.

**表4-2 LRU容量与L3 Miss率对应关系**

| LRU容量（页数） | L3 Miss率 | CXL访问占比 |
|---------------|----------|-----------|
| 512 | 78% | 68% |
| 1024 | 52% | 41% |
| 2048 | 31% | 23% |
| 4096 | 18% | 12% |

### 4.1.3 PEBSTracer硬件采样实现

PEBSTracer基于Linux perf_event子系统采集PEBS硬件事件。初始化时构造`perf_event_attr`结构体，监听MEM_LOAD_RETIRED.L3_MISS事件（Intel编码0x20D1），设置`precise_ip=2`（最高精度级别），通过mmap将内核PEBS缓冲区映射到用户空间：

```cpp
struct perf_event_attr attr = {};
attr.type = PERF_TYPE_RAW;
attr.config = 0x20D1;           // MEM_LOAD_RETIRED.L3_MISS
attr.precise_ip = 2;
attr.sample_period = 1000;
attr.sample_type = PERF_SAMPLE_ADDR | PERF_SAMPLE_WEIGHT | PERF_SAMPLE_DATA_SRC;

int fd = syscall(__NR_perf_event_open, &attr, pid, -1, -1, 0);
void* buf = mmap(NULL, buf_size, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
```

`collect_samples()`调用时，PEBSTracer读取Ring Buffer中未处理的记录，解析`data_src`字段的bit[4:0]判断数据来源，提取虚拟地址、物理地址和延迟周期数，封装为`MemoryAccessEvent`返回。整个采样链路完全在用户态完成，对被监控进程的性能影响通常低于10%.

### 4.1.4 Trace-Driven模式支持

Trace-Driven模式通过回放预录制的访问轨迹文件驱动仿真，适合需要精确复现特定负载行为的科研场景。系统支持简化CSV格式（`timestamp,vaddr,paddr,type,latency_cycles`）和Linux perf mem原始格式两种输入。以`perf mem record`录制的业务程序Trace可直接加载，TraceDrivenTracer按Trace中的时间戳间隔控制事件发射速率，支持1x/2x/4x倍速回放，`collect_samples()`每次返回当前Epoch时间窗口内的事件片段.

## 4.2 CXL拓扑建模与路径计算子系统

### 4.2.1 拓扑图数据结构设计

CXL拓扑具有层次化、参数化、动态可修改三个核心特征。综合这些特征，本系统选择邻接表表示图，以`std::map<string, TopologyNode>`和`std::map<pair<string,string>, TopologyEdge>`分别存储节点和边，以字符串ID作为索引键.

`TopologyNode`使用联合体存储不同类型节点的特有属性：

```cpp
struct TopologyNode {
    std::string id;
    ComponentType type;  // ROOT_COMPLEX | SWITCH | CXL_DEVICE
    union {
        struct { uint64_t dram_size_gb; double dram_latency_ns; } root_complex;
        struct { double processing_latency_ns; int num_ports; } switch_data;
        struct { uint64_t capacity_gb; double base_latency_ns; } cxl_device;
    } data;
};
```

`TopologyEdge`存储链路的静态规格参数和运行时动态负载：

```cpp
struct TopologyEdge {
    std::string from_id, to_id;
    int pcie_generation;
    int link_width;
    double bandwidth_gbps;
    double latency_ns;
    double current_load;  // 当前利用率 [0.0, 1.0]，每Epoch更新
};
```

`current_load`字段由TimingAnalyzer在每个Epoch内累加访问字节数后换算，是拥塞延迟计算的输入，体现了静态配置与动态运行时状态在同一数据结构中的统一管理.

### 4.2.2 拓扑配置与管理

拓扑通过JSON文件描述，ConfigParser负责解析并填充`CXLSimConfig`结构体，再由`TopologyGraph::build_from_config()`构建图模型。ConfigParser在加载时执行三层验证：字段存在性检查（必填字段是否缺失）、值域合法性检查（如带宽不超过该代数位宽的理论上限）、拓扑连通性检查（每个Device是否可从某个RC可达）。任何验证失败均在加载阶段报错，并给出定位到具体字段的错误信息，避免用户带着非法配置运行仿真.

动态拓扑重构方面，用户在图形编辑器中修改拓扑后，TopologyGraph的`dirty_cache_`标志被置位，下一次路径查询时触发全量预计算重建路径缓存，保证缓存与当前拓扑的一致性.

### 4.2.3 最短路径发现与预计算缓存

系统采用Dijkstra算法实现最短延迟路径查找，以边的`latency_ns`为权重，Switch节点的`processing_latency_ns`在访问该节点时叠加，符合Switch延迟属于"进入节点时固定开销"的物理语义。算法使用`std::priority_queue`作为最小堆，提前终止优化（到达目标节点即退出）将平均比较次数降低约40%：

```cpp
TopologyPath TopologyGraph::dijkstra(const string& from, const string& to) {
    unordered_map<string, double> dist;
    unordered_map<string, string> prev;
    for (auto& [id, _] : nodes_) dist[id] = 1e18;
    dist[from] = 0.0;

    priority_queue<pair<double,string>, vector<pair<double,string>>, greater<>> pq;
    pq.push({0.0, from});

    while (!pq.empty()) {
        auto [d, u] = pq.top(); pq.pop();
        if (u == to) break;
        if (d > dist[u]) continue;
        for (auto& nbr : get_neighbors(u)) {
            auto& edge = edges_.at({u, nbr});
            double nd = dist[u] + edge.latency_ns;
            if (nodes_.at(nbr).type == ComponentType::SWITCH)
                nd += nodes_.at(nbr).data.switch_data.processing_latency_ns;
            if (nd < dist[nbr]) { dist[nbr] = nd; prev[nbr] = u; pq.push({nd, nbr}); }
        }
    }
    // 回溯路径并返回...
}
```

在仿真稳定运行阶段，系统预计算所有（RC, Device）对的路径并存入哈希表，运行时查询退化为O(1)哈希查找：

```cpp
unordered_map<string, TopologyPath> path_cache_;  // key: "RC0->DEV2"

void TopologyGraph::precompute_paths() {
    for (auto& rc : get_root_complexes())
        for (auto& dev : get_cxl_devices())
            path_cache_[rc->id + "->" + dev->id] = dijkstra(rc->id, dev->id);
    dirty_cache_ = false;
}
```

对于21节点拓扑，预计算总耗时约0.3 ms，而运行时单次查询降至约20 ns，加速比超过150倍.

### 4.2.4 内存地址映射策略

地址映射策略决定物理地址落在哪个CXL设备的地址区间，系统实现了三种策略，通过统一的`MemoryPolicy`接口切换，仿真过程中可热切换而无需重启.

**First-Touch策略**将每个新分配页面映射到发起分配的CPU所在节点，符合Linux NUMA默认行为，具有良好的空间局部性，但在多核并发场景下容易导致设备负载不均.**Interleave策略**将第N个页面映射到第（N mod D）个设备，实现轮询式负载均衡，各设备利用率趋于均匀，适合随机访问密集型工作负载，但破坏空间局部性.**Tiering策略**维护每个页面的访问热度计数，每隔若干Epoch将热页迁移至延迟更低的本地DRAM，将冷页迁移至CXL设备，动态适应应用访问模式变化，单次页面迁移的开销约为5至6 μs（4 KB数据拷贝+TLB Shootdown）。

**表4-3 三种映射策略特性对比**

| 策略 | 负载均衡 | 空间局部性 | 动态自适应 | 实现复杂度 |
|-----|--------|----------|----------|---------|
| First-Touch | 差 | 优 | 无 | 低 |
| Interleave | 优 | 差 | 无 | 低 |
| Tiering | 中 | 优 | 优 | 高 |

## 4.3 时序分析与延迟建模子系统

### 4.3.1 Epoch驱动执行模型

TimingAnalyzer是整个后端的调度中枢，采用Epoch驱动（Epoch-Driven）执行模型。Epoch是一个固定时长的时间片（默认10 ms），仿真以Epoch为最小统计粒度推进。10 ms足够容纳数千次内存访问事件，统计结果具有代表性；同时比UI刷新周期（33 ms）短，能支持足够细粒度的实时监控.

仿真主循环在独立的QThread工作线程中运行，避免阻塞Qt的UI事件循环：

```cpp
void TimingAnalyzer::run_epoch() {
    topology_->reset_link_loads();
    auto events = tracer_->collect_samples();
    EpochStats stats;
    stats.total_accesses = events.size();

    for (auto& ev : events) {
        if (!ev.is_l3_miss) { stats.local_dram_accesses++; continue; }
        string dev_id = address_mapping_.lookup(ev.physical_addr);
        if (dev_id.empty()) { stats.local_dram_accesses++; continue; }
        auto breakdown = latency_model_->calculate(dev_id, *topology_);
        stats.cxl_accesses++;
        latencies_.push_back(breakdown.total_ns);
        auto path = topology_->get_cached_path("RC0", dev_id);
        topology_->update_link_load(path, 64);
    }
    compute_percentiles(stats);
    stats.epoch_number = epoch_counter_++;
    emit statsUpdated(stats);
}
```

主循环通过QTimer以10 ms间隔触发`run_epoch()`，计算完成后通过Qt信号`statsUpdated`向前端广播`EpochStats`，前端槽函数在UI线程中安全地刷新显示，无需手动加锁.

### 4.3.2 多因素延迟计算模型

LatencyModel采用加法分解模型，将CXL内存访问的端到端延迟分解为五个可独立建模的分量：

$$\text{Latency}_\text{total} = L_\text{device} + L_\text{link} + L_\text{switch} + L_\text{protocol} + L_\text{congestion}$$

这种分解的价值不仅在于计算，更在于可解释性——系统可以向用户呈现每次访问的延迟来自哪里，帮助定位性能瓶颈.

**$L_\text{device}$（设备基础延迟）** 直接取自`CXLDeviceConfig::base_latency_ns`，代表设备侧内存控制器处理加DRAM介质访问的固定延迟，典型值DDR5设备约170 ns.

**$L_\text{link}$（链路传播延迟）** 等于路径上所有PCIe链路`latency_ns`之和，单跳约10至20 ns.

**$L_\text{switch}$（Switch处理延迟）** 对路径经过的每个Switch节点，叠加其`SwitchConfig::latency_ns`（默认40 ns/跳）。

**$L_\text{protocol}$（协议开销）** 固定比例项，取`base_latency × 0.05`，对应CXL.mem的Flit编解码、CRC校验等固定处理时间，约8至10 ns.

**$L_\text{congestion}$（拥塞惩罚）** 采用M/M/1排队论模型计算，取路径上利用率最高的瓶颈链路负载ρ，平均排队延迟为：

$$L_q = \frac{\rho}{\mu(1-\rho)}$$

其中μ为链路服务率。当ρ<0.5时拥塞惩罚可忽略，ρ超过0.9后惩罚急剧增大，准确描述了链路近饱和时的延迟骤增现象。实现中将ρ上限截断在0.99以避免除零，并在日志中输出链路过载警告.

此外，系统通过50 ns时间窗口检测并发访问组（MLP优化）：时间戳差异在50 ns内、目标同一设备的访问视为一个并发组，组内第一个访问计算完整延迟，后续访问仅计算增量Flit传输时间。对于4个并发访问，MLP优化可将总延迟从约1040 ns降至约260 ns.

### 4.3.3 Epoch统计指标计算

每个Epoch结束时，TimingAnalyzer对本Epoch收集的所有CXL访问延迟值进行统计聚合，产出`EpochStats`：

```cpp
struct EpochStats {
    uint64_t total_accesses;
    uint64_t l3_misses;
    uint64_t cxl_accesses;
    uint64_t local_dram_accesses;
    double avg_latency_ns;
    double p95_latency_ns;
    double p99_latency_ns;
    double queuing_delay_ns;
    double link_utilization_pct;
    uint64_t epoch_number;
};
```

P95/P99分位数通过对本Epoch延迟值列表排序后取对应百分位实现，时间复杂度O(N log N)，N为单Epoch访问次数（通常1000至10000），耗时约0.1至1 ms，在Epoch时长范围内可接受。链路利用率取所有链路利用率的最大值，反映系统的瓶颈链路状态.

### 4.3.4 基准管理与A/B实验

实验管理模块提供"基准固定→参数修改→对比分析"的A/B实验工作流。用户点击"固定基准"后，系统将当前`EpochStats`快照保存为`BenchmarkBaseline`，之后修改拓扑或参数重新运行仿真，ComparisonWidget实时计算当前指标与基准的差值和百分比变化，延迟上升标注红色，延迟下降标注绿色。实验数据以CSV格式导出时，包含所有Epoch的完整`EpochStats`记录，首行为列名，方便pandas、MATLAB等工具直接读取.

## 4.4 Qt图形前端设计与实现

### 4.4.1 前端整体架构

前端基于Qt6框架，采用MVC设计模式：后端TimingAnalyzer和TopologyGraph作为Model，各Widget作为View，MainWindow作为Controller协调两者。整体界面布局参考VS Code的侧边栏+主区域+右侧面板三栏结构，用户无需阅读文档即可理解导航逻辑.

组件层次如下：
- **MainWindow**：顶层容器，持有SidebarWidget、QStackedWidget和MetricsPanel
- **SidebarWidget**（宽48 px）：5个图标按钮，点击切换中央页面
- **QStackedWidget**：管理拓扑编辑器、配置管理、工作负载配置、基准测试四个功能页
- **MetricsPanel**（宽280 px，固定右侧）：实时性能监控，在任意页面下始终可见

前后端通信完全通过Qt信号槽实现，TimingAnalyzer在工作线程中`emit statsUpdated(stats)`，MainWindow接收后分发给MetricsPanel等订阅者。Qt的跨线程队列连接自动保证线程安全，这是选择Qt而非原生多线程的关键优势之一.

### 4.4.2 拓扑可视化编辑器

拓扑编辑器基于QGraphicsView框架实现。选择此框架而非直接在QWidget上绘制的原因有三：QGraphicsScene内置了场景坐标与视图坐标的分离，便于实现缩放和平移；QGraphicsItem提供了拖拽、选择等交互基础设施，只需重写`paint()`实现自定义外观；对数十个图形项的批量渲染性能显著优于在`paintEvent`中手动逐一绘制.

三类节点分别继承QGraphicsItem，重写`paint()`实现差异化外观：Root Complex绘制为蓝色圆角矩形，CXL Switch绘制为橙色菱形，CXL Device绘制为绿色圆角矩形。节点继承`ItemIsMovable`标志，拖动时通过重写`itemChange()`回调实时更新相连边的路径坐标。连接线使用`QGraphicsPathItem`，路径形状为三次贝塞尔曲线，仿真运行时连接线上叠加颜色指示条（低于50%绿色、50%至80%黄色、超过80%红色），直观反映链路拥塞状态.

为避免节点重叠与链路交叉，编辑器提供一键"自动布局"功能，采用力导向图（Force-Directed Graph）算法：节点间存在库仑斥力，连接节点间存在胡克弹簧引力，迭代100次后收敛，使节点分布均匀，链路长度趋于一致，特别适合CXL拓扑的树形和稀疏图结构.

### 4.4.3 配置树管理界面

配置树管理界面使用QTreeWidget以层次化形式展示当前拓扑的完整配置参数，树的根节点对应整个仿真配置，一级子节点对应各组件类型，二级子节点对应各具体实例，三级叶节点对应各参数字段（可内联编辑）。用户双击任一参数值节点即可进入行内编辑模式，失去焦点时自动验证输入格式和值域，验证失败字段以红色高亮并显示tooltip错误提示，验证通过则立即更新内存中的配置对象并通知TopologyGraph置位`dirty_cache_`标志.

### 4.4.4 实时性能监控面板

MetricsPanel固定显示在界面右侧，是"仿真状态的持续感知窗口"。面板分为三个区域：顶部LCD仪表盘大字显示当前Epoch平均延迟；中部数据表格按分组展示各项`EpochStats`指标；底部`RealTimeChartWidget`嵌入QChart折线图，展示平均延迟和P95延迟随Epoch编号的动态趋势，保留最近100个Epoch的数据点（滚动窗口）。

为防止高频更新（每10 ms一个Epoch）导致UI线程过载，MetricsPanel内部实现节流机制：收到`statsUpdated`信号后，若距上次UI刷新时间不足33 ms（对应30 FPS），则将stats暂存到`pending_stats_`，待下一次刷新定时器触发时一并更新，保证前端始终流畅不卡顿.

### 4.4.5 基准测试与对比界面

BenchmarkPage包含BenchmarkWidget（配置与控制区）和ComparisonWidget（对比展示区）两个子组件。用户点击"固定基准"后，界面以灰色背景展示基准数值，用户随后修改拓扑重新运行仿真，ComparisonWidget每行显示"当前值 | 基准值 | 差异%"，差异为正以红色标注向上箭头，为负以绿色标注向下箭头。这种视觉差异呈现将传统"记录A组→记录B组→手动计算差值→判断好坏"的四步操作压缩为"固定基准→改配置→直接看颜色"的三步操作，显著提升了迭代实验的效率.

## 4.5 本章小结

本章按照数据流方向，系统介绍了CXLMemSim四个子系统的设计与实现。追踪子系统通过ITracer抽象接口和TracerFactory自动探测，实现了MockTracer、PEBSTracer和TraceDriven三种后端的无缝切换，MockTracer中Zipf热点分布和LRU缓存仿真保证了合成流量的统计代表性。拓扑建模子系统将CXL系统形式化为有向加权图，Dijkstra路径算法结合预计算缓存将运行时路径查询降至O(1)，三种地址映射策略覆盖了从简单到复杂的多种研究场景。延迟计算模型的五因素分解框架提供了可解释的延迟细化分析，M/M/1拥塞建模准确描述了链路近饱和时的延迟骤增行为，MLP并发优化进一步提升了建模精度。Qt前端通过MVC架构实现了前后端的彻底解耦，拓扑编辑器、实时监控面板、基准对比组件共同构成了覆盖"构建→仿真→对比→导出"完整科研工作流的一站式操作界面。下一章将通过系统性的实验对上述设计的功能正确性和性能建模精度进行全面验证.
