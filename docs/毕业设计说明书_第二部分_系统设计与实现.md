# 3  系统设计与实现

## 3.1  引言

本章详细介绍CXLMemSim各个模块的设计思路与具体实现。整个系统在架构上分为三个层次：最底层是公共基础模块，包括配置文件解析和数据结构定义；中间层是后端仿真引擎，由追踪子系统、拓扑建模子系统、延迟计算模型和仿真协调器组成；最上层是基于Qt 6的图形前端，负责与用户交互并实时展示仿真结果。各层之间的通信遵循清晰的接口约定，后端与前端之间通过Qt信号槽机制异步传递数据，保证了仿真计算不会阻塞界面响应。

图3-1展示了系统的整体架构关系。从图中可以看出，Qt前端的各个功能页面并不直接调用后端的计算函数，而是通过MainWindow这个中枢控制器与TimingAnalyzer（时序分析器）通信，TimingAnalyzer再分别调度Tracer、TopologyGraph和LatencyModel三个核心子系统。这种设计的好处是，即便后端某个子系统的实现发生变化（比如换用不同的Tracer后端），只要接口契约不变，前端代码就完全不需要修改。

本章按照从底层到上层的顺序展开，依次介绍配置管理、追踪子系统、拓扑建模、延迟计算模型、仿真协调器，最后介绍Qt前端的各个功能模块。

## 3.2  系统需求分析

**3.2.1  功能性需求**

在充分调研现有CXL仿真工具的基础上，结合潜在用户群体（体系结构方向的研究生、数据中心基础设施工程师、高校课程教学）的实际需求，本课题确定了以下主要功能性需求。

第一，系统需要支持灵活的CXL拓扑配置，用户应该能够描述包含多台主机（Root Complex）、多级CXL Switch和多个CXL内存设备的复杂拓扑，并能为每条链路配置PCIe规格（代数和位宽）、带宽、延迟等参数，为每个设备配置容量、访问延迟等属性。考虑到手工编写JSON配置文件容易出错且不够直观，系统还应提供可视化的图形拓扑编辑器，让用户通过拖拽操作完成拓扑构建，并能将编辑结果保存和加载。

第二，系统需要在不依赖物理CXL硬件的条件下能够正常运行，这意味着必须提供合理的内存访问流量模拟手段。对于条件允许的用户，还应该支持接入真实的硬件采样数据。两种模式之间的切换应该对用户透明，不需要修改其他任何配置。

第三，系统需要对CXL内存访问的延迟进行准确建模，计算结果应该区分延迟的不同来源，包括设备延迟、链路延迟、Switch转发延迟、协议开销以及拥塞引入的排队延迟，以便用户能够诊断性能瓶颈。

第四，系统需要提供实时的性能监控视图，在仿真运行过程中以Epoch为单位持续更新并展示平均访问延迟、P95/P99尾延迟、L3 Miss率、链路利用率、CXL访问占比等核心指标，并以折线图等可视化形式展示指标的动态变化趋势。

第五，系统需要支持基准（Baseline）管理和多配置对比实验。用户可以将某一时刻的性能快照保存为基准，然后修改拓扑或参数配置后重新运行仿真，系统自动计算当前结果与基准的差异并标注改善或恶化情况，从而支持"更改一个变量、观察性能变化"这种典型的科研实验模式。

第六，系统需要支持实验数据的导出，用户应该能够将仿真产生的Epoch级统计数据导出为CSV或JSON格式，以便使用Python、R等工具进行进一步的分析和绘图。

**3.2.2  非功能性需求**

在非功能性方面，系统需要满足以下约束。

运行环境方面，系统需要能在Ubuntu 20.04及以上版本的Linux系统中正常运行，优先保证在VMware Workstation、VirtualBox等常见虚拟机环境中的可用性，不强制要求物理CXL硬件。

性能方面，系统自身的运行开销应该控制在可接受范围内，在MockTracer模式下CPU占用率不超过15%，内存占用不超过512MB，仿真帧率（每秒完成的Epoch数）能够支持实时交互级别的流畅度。

可维护性方面，各模块之间的依赖关系应该尽量简单，接口定义清晰，新增一种Tracer后端不应该需要修改现有模块的代码。

## 3.3  系统总体设计

**3.3.1  分层架构设计**

CXLMemSim采用三层架构，如图3-1所示，从下往上依次为公共基础层、后端仿真引擎层和Qt图形前端层。

公共基础层位于`common/`目录，包含ConfigParser（配置文件解析器）、各类配置数据结构（`LinkSpec`、`RootComplexConfig`、`SwitchConfig`、`CXLDeviceConfig`等）以及公共工具函数。这一层不依赖Qt，也不依赖Linux特定系统调用，保证了跨平台的可移植性。

后端仿真引擎层位于`backend/`目录，包含三个相对独立的子系统：Tracer追踪子系统（`backend/tracer/`）、TopologyGraph拓扑图管理器（`backend/topology/`）和LatencyModel延迟计算模型（`backend/analyzer/`）。这三个子系统由TimingAnalyzer（`backend/analyzer/timing_analyzer.h`）统一协调，TimingAnalyzer以Epoch为驱动周期，在每个Epoch内完成采集事件、映射地址、计算延迟、更新统计的完整循环。后端整体作为一个独立的C++库（`cxlmemsim_backend`）编译，在CMakeLists.txt中单独定义target。

Qt图形前端层位于`frontend/`目录，以MainWindow为中枢，包含六类功能页面和若干可复用的Widget组件，具体结构在3.7节详细介绍。前端通过实例化TimingAnalyzer并与之建立信号槽连接来驱动后端运行，后端产生的每个Epoch统计数据通过信号发射到前端，由相应的槽函数更新UI显示。

```
目录结构：
CXLMemSim/
├── common/include/      # 公共头文件（config_parser.h等）
├── backend/
│   ├── include/tracer/  # tracer_interface.h
│   ├── include/topology/# topology_graph.h
│   ├── include/analyzer/# latency_model.h, timing_analyzer.h
│   └── src/             # 各模块实现文件
├── frontend/
│   ├── mainwindow.h/cpp
│   └── widgets/         # 各功能Widget
├── configs/examples/    # 示例JSON配置文件
└── tests/               # Google Test单元测试
```

**3.3.2  关键数据流**

系统的核心数据流如下：用户在前端拓扑编辑器中构建拓扑，编辑结果序列化为CXLSimConfig结构体并传入后端的TopologyGraph，TopologyGraph据此构建有向加权图。用户点击"开始仿真"后，MainWindow调用TimingAnalyzer的start()方法，后者启动一个工作线程（QThread）运行仿真循环：每个Epoch内，TimingAnalyzer先调用ITracer::collect_samples()收集一批MemoryAccessEvent，再遍历每个事件，通过地址到设备的映射表（AddressMapping列表）确定该访问落在哪个CXL设备上，然后调用LatencyModel::calculate_latency()计算该访问的各分项延迟，同时调用TopologyGraph::update_link_load()更新路径上各链路的当前负载。Epoch结束时，TimingAnalyzer汇总所有访问的延迟数据，计算均值、P95、P99等统计量，打包成EpochStats结构体，通过Qt信号statsUpdated(EpochStats)发射给前端。前端的MetricsPanel槽函数接收到信号后，更新各个指标标签和实时图表。

## 3.4  配置管理模块

**3.4.1  配置数据结构设计**

配置管理是整个系统的数据源头，其设计的合理性直接影响到拓扑描述能力和扩展灵活性。CXLMemSim使用JSON格式的配置文件描述仿真场景，配置文件的内容通过ConfigParser解析后填充到一套C++结构体中，供后端各模块使用。

链路规格用LinkSpec结构体描述，包含PCIe代数（generation字段，取值4或5）、链路位宽（width字段，取值8或16）、带宽（bandwidth_gbps，单位GB/s）和基础延迟（latency_ns，单位ns）。代际和位宽的组合决定了理论最大带宽，例如PCIe 5.0 x16的单向带宽约为64GB/s，PCIe 5.0 x8约为32GB/s。

```cpp
struct LinkSpec {
    std::string type;       // 如 "Gen5x16"
    int generation;         // 4 或 5
    int width;              // 8 或 16
    double bandwidth_gbps;  // 最大带宽 GB/s
    double latency_ns;      // 单程传播延迟 ns
};
```

CXL设备用CXLDeviceConfig描述，除了基本的容量、链路规格信息外，还区分了base_latency_ns（设备整体基础延迟，包含控制器处理时间）和media_latency_ns（介质访问延迟，对DRAM类型约90ns），这种区分使得系统能够模拟DRAM、NVM等不同介质类型的CXL设备之间的延迟差异。supports_hdm和supports_coherency两个布尔字段用于标记设备是否支持HDM（Host-managed Device Memory）模式和设备侧一致性，当前版本主要面向Type 3（纯内存扩展）设备，因此这两个字段主要用于信息记录和未来功能扩展。

拓扑连接关系用ConnectionConfig结构体描述，包含源节点ID、目标节点ID以及该链路的LinkSpec。这种显式列举连接关系的方式比邻接矩阵更灵活，容易表达非均匀拓扑（不同链路参数不同的情况）。

## 3.5  内存访问追踪子系统

**3.5.1  抽象接口设计**

追踪子系统的设计目标是让系统能够在不同的运行环境下无缝切换采样后端，同时对上层的TimingAnalyzer屏蔽后端实现细节。为此，本课题设计了ITracer抽象基类，定义了所有追踪器实现必须提供的接口契约：

```cpp
class ITracer {
public:
    virtual ~ITracer() = default;
    virtual bool initialize(pid_t target_pid) = 0;
    virtual bool start() = 0;
    virtual void stop() = 0;
    virtual std::vector<MemoryAccessEvent> collect_samples() = 0;
    virtual std::string name() const = 0;
    virtual bool supports_precise_address() const = 0;
    virtual bool is_running() const = 0;
};
```

其中collect_samples()是最核心的方法，它在每个Epoch被调用一次，返回该时间片内采集到的所有内存访问事件列表。MemoryAccessEvent结构体记录了单次内存访问的完整信息，包括时间戳（timestamp_ns）、虚拟地址（virtual_addr）、物理地址（physical_addr，部分后端可能无法提供）、线程ID（tid）、CPU核编号（cpu）、是否为Load操作（is_load）、是否触发了L3 Miss（is_l3_miss）等字段。特别值得一提的是data_source字段，它保存Intel PEBS记录中的Data Source位域原始值，供PEBSTracer解码访问来源（L1 Hit/L2 Hit/L3 Miss/远端DRAM等）时使用。

TracerFactory类负责Tracer实例的创建和环境探测，其静态方法detect_best_tracer()会按优先级依次检测当前环境：先检查是否存在`/proc/sys/kernel/perf_event_paranoid`且其值允许用户空间访问PMU（通常要求该值≤1），如果满足则选择PEBSTracer；再检查是否存在预录制的Trace文件（通过命令行参数指定路径）；都不满足的情况下默认使用MockTracer。这种自动探测机制使得同一个可执行程序可以在虚拟机和物理机上分别选择最合适的后端，用户无需手动指定。

**3.5.2  MockTracer的实现**

MockTracer是面向虚拟机环境的合成流量生成器，其核心任务是产生在统计特性上接近真实应用的访问序列。实现上，MockTracer维护了一个固定大小的"虚拟地址空间"（默认32GB），将这个空间切分为若干4KB的页面，并使用Zipf分布生成每次访问的目标页面编号。

Zipf分布的采样使用逆变换法（Inverse Transform Sampling）实现：预先计算Zipf累积分布函数的逆映射表（大小为页面总数），然后每次采样时生成一个均匀随机数并在表中二分查找对应的页面编号。对于包含几十万个页面的大地址空间，直接构建完整逆映射表内存开销过大，实际实现中采用了分段近似方法，将热门的前1000个页面精确建表，其余页面按区间均匀近似，在保证热点特征的前提下将内存开销控制在可接受范围内。

L3 Miss的模拟通过内置的LRU缓存实现。MockTracer维护一个容量可配置的LRU缓存（默认对应16MB的LLC，即4096个页面槽位），每次生成访问事件时查询这个缓存：缓存命中则标记is_l3_miss为false，缓存未命中则标记为true并将该页面加入缓存（同时淘汰最久未使用的页面）。LRU缓存的容量参数直接决定了仿真中的L3 Miss率，用户可以通过调整这个参数来模拟不同程度的缓存压力。在Zipf分布参数α=0.99、LRU容量4096页的默认配置下，MockTracer产生的L3 Miss率约在35%至45%之间，与典型的内存密集型应用（如图数据库、大型矩阵运算）的实测L3 Miss率比较接近。

**3.5.3  PEBSTracer的实现**

PEBSTracer针对物理机环境，通过Linux perf_event子系统采集PEBS硬件事件。初始化阶段，PEBSTracer构造perf_event_attr结构体，将type字段设置为PERF_TYPE_RAW，将config字段设置为`MEM_UOPS_RETIRED.ALL_LOADS`事件的原始编码值（Skylake及后续微架构上该事件编码为0x81D0），并设置precise_ip字段为2（请求PEBS精确采样），sample_type字段指定需要记录IP、地址、Data Source等信息。调用perf_event_open后，通过mmap将内核PEBS缓冲区映射到用户空间，大小为64个内存页（256KB），足以容纳约3000条PEBS记录而不溢出。

collect_samples()调用时，PEBSTracer读取PEBS缓冲区的头尾指针，遍历未读取的PEBS记录，解析每条记录中的采样地址和Data Source字段。Data Source字段的bit 5:0编码了数据来源，当这几位的值对应"L3 Miss"时（具体编码值因微架构而异，Skylake上为0b001111），将对应的事件标记为L3 Miss，填充到MemoryAccessEvent结构体后返回给调用方。这种基于硬件计数器的采样方式overhead极低，对被监控应用的性能影响通常在1%至3%以内，是目前在物理机上采集精确内存访问事件的最实用方法。

## 3.6  CXL拓扑建模子系统

**3.6.1  图模型的设计**

TopologyGraph类将CXL拓扑表示为一个有向加权图，图的顶点（TopologyNode）对应拓扑中的组件实体，图的边（TopologyEdge）对应PCIe/CXL链路。顶点有三种类型（ComponentType枚举：ROOT_COMPLEX、SWITCH、CXL_DEVICE），每种类型携带不同的属性数据，使用union实现（Root Complex存储本地DRAM容量和延迟，Switch存储端口数和转发延迟，CXL Device存储容量、基础延迟和一致性标志）。边存储链路的PCIe代数、位宽、带宽上限、单程延迟以及当前运行时负载（current_load字段，取值0.0至1.0）。

内部存储采用两个`std::map`：`nodes_`以节点ID（字符串）为键映射到TopologyNode，`edges_`以（源节点ID, 目标节点ID）的pair为键映射到TopologyEdge。选用std::map而非std::unordered_map，一方面是因为拓扑规模通常很小（节点数一般不超过几十个），有序map在查找性能上与哈希map差异可以忽略；另一方面，有序map的迭代顺序是确定的，便于调试时输出一致的拓扑结构信息。

**3.6.2  路径查找算法**

TopologyGraph实现了基于Dijkstra算法的最短延迟路径查找，接口为find_path(from_id, to_id)，返回一个TopologyPath对象，包含路径经过的节点序列、边序列、路径总延迟和路径瓶颈带宽。

Dijkstra的实现使用了`std::priority_queue`作为最小优先队列，以节点到源节点的当前最小延迟为优先级（最小值优先）。边的权重取TopologyEdge::latency_ns字段，即链路的单程传播延迟；对于经过Switch的路径，Switch节点的转发延迟通过在节点处理时累加SwitchConfig::latency_ns来计入，而不是放在边权重里，这样更符合物理实际（Switch延迟是进入端口时的固定开销，与后续选择哪个下行端口无关）。

对于频繁查询同一拓扑的场景（如仿真运行期间每个Epoch需要处理数千次路径查询），每次都从头运行Dijkstra是不必要的开销。CXLMemSim在TopologyGraph被修改后（比如用户在编辑器中添加了节点或修改了参数）会设置一个脏标志（dirty_cache_），并在下一次路径查询时触发全量预计算：对拓扑中的每一对（RC, CXL Device）运行一次Dijkstra，将结果缓存在`path_cache_`哈希表中。这样在仿真稳定运行阶段，路径查询只需要一次哈希表查找，时间复杂度从O((V+E) log V)降到O(1)，对于21节点（1RC+4SW+16DEV）规模的拓扑，仿真阶段的路径查询耗时从约8.5μs降到了0.02μs左右。

**3.6.3  链路负载追踪**

仿真运行时，每处理一个被路由到CXL设备的访问事件，TimingAnalyzer会调用TopologyGraph::update_link_load()，将访问请求和响应数据量（读请求64字节数据，写请求也是64字节）累加到对应路径上每条链路的当前负载中。为了将瞬时访问量转换为带宽利用率，TopologyGraph在每个Epoch开始时清零所有链路的负载计数，在Epoch结束时统一将累计字节数除以Epoch时长（默认10ms），得到本Epoch内各链路的平均带宽使用量，再除以链路最大带宽上限，得到利用率百分比（0至100%）。这个利用率数值被反馈给LatencyModel用于计算拥塞惩罚延迟，形成一个"访问量→链路负载→拥塞延迟→统计数据"的闭环。

## 3.7  延迟计算模型

**3.7.1  多因素延迟分解**

LatencyModel类是仿真精度的核心所在，它将一次CXL内存访问的端到端延迟分解为五个组成部分，每部分的计算方法如下。

**设备基础延迟（base_latency_ns）** 直接取自配置文件中该设备的CXLDeviceConfig::base_latency_ns字段，代表CXL设备在无拥塞情况下从收到请求到发出响应的固定延迟，包含设备侧内存控制器处理时间和DRAM或NVM介质的访问时间。默认值170ns对应一款典型的DDR5-4800 CXL内存模块的实测延迟（参考Samsung CXL DRAM白皮书数据）。

**链路传播延迟（link_latency_ns）** 通过TopologyPath::total_latency_ns获得，这个值在路径预计算阶段就已经累加了路径上所有链路的latency_ns。对于PCIe 5.0短距离（服务器内部）连接，单跳链路延迟通常在10至20ns之间，主要由PHY层串化/解串化（SerDes）延迟决定。

**Switch处理延迟（switch_latency_ns）** 对于路径上经过的每一个Switch节点，叠加其SwitchConfig::latency_ns。当前默认值设置为40ns，参考了CXL Consortium规范中对Switch端口转发延迟的估算值。

**协议开销（protocol_overhead_ns）** 固定值，取LatencyModelParams::protocol_overhead_ratio × base_latency_ns，默认比例为0.05（即5%），对170ns基础延迟来说约8.5ns，主要对应CXL协议栈的Flit编解码和信用（Credit）流控处理时间。

**拥塞惩罚（congestion_penalty_ns）** 这是变化最动态的分量，需要根据链路的当前利用率实时计算。实现中取路径上利用率最高的那条链路（瓶颈链路）的负载ρ，然后按M/M/1公式计算：

$$L_q = \frac{1}{\mu} \cdot \frac{\rho}{1-\rho}$$

其中μ = bandwidth_gbps × 10⁹ / 64（每秒能处理的64字节访问次数，近似为服务率），ρ是当前利用率。当ρ小于0.5时，拥塞惩罚通常在几纳秒以内，可以忽略；当ρ超过0.9后，惩罚延迟会急剧增大，在ρ=0.95时理论值约为服务时间的19倍，这正是CXL链路接近饱和时用户体验到的延迟剧烈抖动现象的模型描述。

最终，总延迟为以上五项之和，并存储到LatencyBreakdown结构体中，供后续的统计汇总和可视化使用：

```cpp
struct LatencyBreakdown {
    double base_latency_ns;
    double link_latency_ns;
    double switch_latency_ns;
    double protocol_overhead_ns;
    double congestion_penalty_ns;
    double total_ns;           // 以上各项之和
};
```

**3.7.2  MLP优化建模**

MLP（Memory-Level Parallelism，内存级并行）是指现代处理器流水线中同时存在多个未完成的内存访问请求，这些并发的请求可以在内存控制器和内存通道中流水线处理，使得多个访问的总延迟小于各自延迟之和。LatencyModel通过LatencyModelParams::enable_mlp开关控制是否启用MLP优化建模。启用时，对于TimingAnalyzer在同一个时间窗口（50ns以内，接近CPU乱序执行窗口深度）内收集到的多个访问同一设备的事件，LatencyModel会将它们视为并发访问组，只对第一个访问计算完整的端到端延迟，后续访问在前一个访问完成之前到达的部分，其延迟仅计算增量传输时间（约等于单个64字节Flit的传输时间，在PCIe 5.0 x16上约1ns），从而反映流水线并发处理的加速效果。在访问具有良好并发性的工作负载（如大步长顺序访问）场景下，MLP优化可以使仿真统计的平均延迟降低10%至30%，更接近真实处理器的测量值。

## 3.8  仿真协调器

**3.8.1  TimingAnalyzer的工作流程**

TimingAnalyzer是整个后端的调度中枢，它持有ITracer、TopologyGraph和LatencyModel的实例引用，并在工作线程中运行以Epoch为单位的仿真循环。

TimingAnalyzer接收一个TimingAnalyzerConfig配置对象，其中epoch_duration_ms决定每个Epoch的时长（默认10ms），enable_injection标志控制是否实际向目标进程注入延迟（当前版本主要用于统计分析，延迟注入功能为预留接口），verbose_logging控制是否输出详细的逐事件日志。

仿真循环的主体在TimingAnalyzer内部的一个工作线程中运行，主循环结构大致如下：每次循环开始时先重置TopologyGraph的链路负载统计，然后调用ITracer::collect_samples()收集本Epoch的访问事件列表，遍历事件列表时对每个事件执行地址映射查找（通过address_mappings_列表的线性扫描确定物理地址落在哪个CXL设备的地址区间内，如果不在任何CXL地址范围则标记为本地DRAM访问），然后调用LatencyModel::calculate_latency()计算延迟，将结果累加到本Epoch的聚合统计中。等到Epoch时长（由QTimer计时）到期时，计算分位数统计（P95、P99通过对本Epoch所有延迟值排序后取对应百分位实现），构造EpochStats结构体，通过Qt信号statsUpdated发射给前端。

**3.8.2  地址映射管理**

地址映射表（AddressMapping结构体的列表）记录了各CXL设备对应的物理地址区间。在真实系统中，这些映射由BIOS在启动时根据CXL设备描述符自动配置并写入ACPI表（CEDT和SRAT表），操作系统通过读取这些ACPI表识别CXL内存节点。在仿真环境中，地址映射关系由用户在配置文件中显式指定，ConfigParser解析后作为初始化参数传入TimingAnalyzer。

每个AddressMapping包含start_addr、end_addr（64位整数，表示物理地址区间的起止）和device_id（对应TopologyGraph中CXL设备节点的ID），其contains()方法用于判断一个给定地址是否落在该区间内。查询时对映射表做顺序扫描，由于通常只有少量设备（5个以内），顺序扫描的性能完全足够。

## 3.9  Qt图形前端设计与实现

**3.9.1  整体界面布局**

CXLMemSim的图形界面参考了现代IDE（如VS Code）的布局设计：左侧是一个宽约48像素的SidebarWidget，由5个图标按钮组成导航栏，点击不同按钮切换中央区域显示的功能页面；中央区域是QStackedWidget管理的页面栈，包含拓扑编辑器页、配置管理页、工作负载配置页、基准测试页；右侧是固定显示的MetricsPanel（性能指标监控面板），宽约280像素，始终可见，即使在编辑拓扑时用户也能同时看到实时的性能数据。顶部有一个MenuBar，提供文件操作（新建、打开、保存配置）、仿真控制（启动、暂停、停止）和帮助等功能。

SidebarWidget的按钮使用Unicode字符作为图标（避免了图标文件依赖），通过自定义样式表实现深色科技感的视觉风格：默认状态背景透明，悬停时左侧出现蓝色高亮条（`border-left: 2px solid #4FC3F7`）并带轻微的渐变背景，选中状态使用更深的渐变背景和蓝色文字。这种风格与整体深色主题保持一致。

**3.9.2  拓扑编辑器**

拓扑编辑器是前端最复杂的模块，基于Qt的QGraphicsView框架实现。编辑器由以下几个部分组成：左侧的组件面板（ComponentPanel），以列表形式显示可拖拽的组件类型（Root Complex、Switch、CXL Device）；中央的QGraphicsView画布，用于渲染和编辑拓扑图；右侧的属性面板（PropertyPanel），双击任意节点时展开，显示该节点的可编辑参数。

每种组件类型对应一个自定义的QGraphicsItem子类。以CXL设备节点为例，其paint()方法绘制一个橙色圆角矩形，中心显示节点ID文本，四个方向各有一个白色小圆形"锚点"表示可连线的端口。节点继承QGraphicsItem::ItemIsMovable标志，允许用户拖拽移动，移动时通过重写itemChange()回调通知所有与该节点相连的边更新路径坐标。

连接线采用QGraphicsPathItem实现，路径形状为三次贝塞尔曲线：控制点根据源锚点和目标锚点的相对位置动态计算，使得无论两个节点处于什么相对位置，连接线都呈现平滑的弧形，不会出现折线或直线带来的生硬感。连接线上还叠加了一个表示带宽利用率的颜色指示条，在仿真运行时实时更新颜色（绿色表示低利用率，黄色表示中等，红色表示高利用率），让用户一眼就能看出哪条链路正在承受压力。

用户在画布上完成拓扑编辑后，点击"应用到仿真"按钮，编辑器会遍历所有QGraphicsItem，将节点和连线信息序列化为CXLSimConfig结构体，传入后端的TopologyGraph::build_from_config()方法。如果拓扑验证通过（TopologyGraph::validate()返回true），系统提示"配置已更新"；否则以对话框形式显示具体的验证错误（如"DEV0与RC0之间不存在路径"），提示用户修正。

**3.9.3  实时性能监控面板**

MetricsPanel在仿真运行期间持续接收来自TimingAnalyzer的EpochStats信号并更新显示，其界面分为三个区域：顶部的仪表盘区域用QLCDNumber大字显示当前Epoch的平均延迟（单位ns）；中部的数据表格区域按分组（访问统计、延迟统计、带宽统计）显示各项指标的当前值；底部是RealTimeChartWidget，内嵌一个QChart图表，默认显示平均延迟和P95延迟随Epoch编号的变化折线，最多保留最近100个Epoch的数据点（超出后从左侧移除，实现滚动显示效果）。

数值更新时，为了避免数字突变给人带来的视觉不适，延迟类数值使用了QPropertyAnimation做缓动过渡：动画持时300ms，缓动曲线为QEasingCurve::OutQuad，使数值从旧值平滑过渡到新值，而非瞬间跳变。链路利用率通过QProgressBar显示，并根据数值区间动态改变颜色：低于50%显示绿色，50%至80%显示黄色，80%以上显示红色，以直观的颜色编码传达拥塞程度。

**3.9.4  基准测试与对比功能**

BenchmarkPageWidget是专门用于基准管理和A/B对比实验的功能页面。用户可以在仿真运行过程中随时点击"固定基准"按钮，系统将当前EpochStats快照保存为BenchmarkBaseline结构体，同时记录对应的拓扑配置快照。之后用户修改拓扑参数（如增加一级Switch、减少链路带宽等）并重新运行仿真，ComparisonWidget会实时对比当前指标与基准的差异：对于延迟相关指标，差异为负值（延迟降低）时标注绿色并显示向下箭头，差异为正值时标注红色并显示向上箭头；对于带宽利用率指标，逻辑相反（利用率降低通常是好事）。

这种一键式的基准对比功能把传统上需要用户手动记录数据、手动计算差异的工作流程化简为几次点击操作，在面向"改变一个配置参数、观察性能变化"这种典型实验场景时效率显著提高，是CXLMemSim在用户体验上相比命令行工具的重要优势之一。

**3.9.5  配置导出与复现**

为了支持实验的可复现性，CXLMemSim提供了完整的配置和结果导出功能。用户可以通过菜单"File → Export Config"将当前拓扑配置序列化为JSON文件保存，之后可以通过"File → Open Config"重新加载，恢复完全相同的实验环境。仿真产生的Epoch级统计数据可以通过"File → Export Data"导出为CSV文件，文件包含时间戳、Epoch编号、所有EpochStats字段的数值，方便用Python的pandas或R的数据分析库做进一步处理。导出CSV的格式设计参考了Linux perf stat的输出惯例，第一行为列名，后续每行对应一个Epoch，字段之间用逗号分隔，数值字段保留两位小数精度。

## 3.10  本章小结

本章系统介绍了CXLMemSim从底层配置管理到上层图形前端的完整设计与实现。在架构层面，三层分离设计使得各模块可以独立演进，前后端通过Qt信号槽机制保持松耦合。追踪子系统的ITracer抽象接口和TracerFactory自动探测机制，使得系统可以在虚拟机（MockTracer）和物理机（PEBSTracer）环境下无缝切换。拓扑建模子系统将CXL拓扑表示为有向加权图，通过Dijkstra算法和预计算缓存机制实现了高效的路径查询。延迟计算模型的五因素分解框架和M/M/1拥塞建模，在保证计算效率的前提下提供了合理的建模精度。Qt图形前端通过精心设计的交互和实时可视化，显著降低了CXL系统配置和性能分析的操作门槛。下一章将通过具体的测试实验，对系统的功能正确性和性能建模精度进行评估。
