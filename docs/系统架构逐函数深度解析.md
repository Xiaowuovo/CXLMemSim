## 一、核心数据结构

在看任何函数之前，先理解贯穿全系统的三个核心结构体。

### `MemoryAccessEvent`
**文件：** `backend/include/tracer/tracer_interface.h:24`

代表系统中**一次内存访问的完整快照**，是从追踪器流向分析器的最小数据单元。

```cpp
struct MemoryAccessEvent {
    uint64_t timestamp_ns;   // 事件发生的时间戳，纳秒精度，由 steady_clock 提供
    uint64_t virtual_addr;   // 被访问的虚拟地址，64字节对齐（& ~0x3F）
    uint64_t physical_addr;  // 物理地址，PEBS模式下由硬件填充，Mock模式为0
    uint32_t tid;            // 发起访问的线程ID
    uint32_t cpu;            // 运行在哪个CPU核心上
    uint64_t latency_cycles; // 硬件测量的延迟（周期数），Mock模式下是ns近似值
    bool is_load;            // true=读操作，false=写操作（写比读慢约10%）
    bool is_l3_miss;         // 关键标志：只有L3 miss才会走CXL链路
    uint64_t ip;             // 触发该访问的指令地址（仅PEBS模式有效）
    uint32_t data_source;    // Intel PEBS的数据来源编码（区分DRAM/PMem/CXL）
};
```

**为什么 `is_l3_miss` 是关键字段？**  
CXL 内存对 CPU 是透明的，CPU 先查 L1→L2→L3 缓存，只有 L3 也没命中，才会真正发出内存访问请求穿越 PCIe 总线到达 CXL 设备。L3 命中的访问延迟约 30~40ns，CXL 访问延迟约 170~300ns，混在一起统计会严重低估 CXL 延迟。因此分析器**只对 `is_l3_miss=true` 的事件计算 CXL 延迟**。

### `EpochStats`
**文件：** `backend/include/analyzer/timing_analyzer.h:42`

代表**一个 Epoch（默认10ms）内的汇总统计结果**，是从后端传给前端的数据包。

```cpp
struct EpochStats {
    uint64_t total_accesses;         // 本Epoch内总内存访问次数（含L3 hit）
    uint64_t l3_misses;              // L3 miss 次数
    uint64_t cxl_accesses;           // 成功映射到CXL设备的访问次数
    uint64_t local_dram_accesses;    // L3 miss但地址不在CXL范围 → 本地DRAM
    double total_injected_delay_ns;  // 本Epoch实际注入的总延迟（用于验证精度）
    double avg_latency_ns;           // CXL访问平均延迟
    double p95_latency_ns;           // 95%的访问低于此值（评估正常情况）
    double p99_latency_ns;           // 99%的访问低于此值（评估最坏情况/尾延迟）
    double queuing_delay_ns;         // 拥塞排队延迟（avg的10%，简化模型）
    double link_utilization_pct;     // 链路利用率百分比 0~100%
    uint64_t epoch_number;           // 全局单调递增的Epoch编号
};
```

### `AddressMapping`
**文件：** `backend/include/analyzer/timing_analyzer.h:29`

描述**一段虚拟地址空间属于哪个 CXL 设备**，是地址空间分割的基本单元。

```cpp
struct AddressMapping {
    uint64_t start_addr;   // 起始地址（含）
    uint64_t end_addr;     // 结束地址（不含）
    std::string device_id; // 对应的CXL设备ID，如 "CXL_MEM1"

    // 判断地址是否属于本映射范围
    bool contains(uint64_t addr) const {
        return addr >= start_addr && addr < end_addr;
    }
};
```

当前实现中每个 CXL 设备分配 1GB 地址空间，从 `0x100000000`（4GB处）开始，4个设备对应 `4GB~8GB`。

## 二、追踪器层 — 事件来源

### 2.1 `ITracer` 抽象接口
**文件：** `backend/include/tracer/tracer_interface.h:51`

这是一个**纯虚基类（接口）**，定义了所有 Tracer 必须实现的协议。这是典型的策略模式（Strategy Pattern）：`TimingAnalyzer` 只依赖 `ITracer*`，不关心底层是 Mock 还是 PEBS。

| 纯虚函数 | 作用 |
|---------|------|
| `initialize(pid_t)` | 绑定要追踪的进程PID，-1表示全系统追踪 |
| `start()` | 开始采集，PEBS模式下会打开perf_event fd |
| `stop()` | 停止采集，关闭fd、释放mmap缓冲区 |
| `collect_samples()` | **核心接口**：返回本Epoch收集到的事件批次 |
| `name()` | 返回实现名称，用于日志显示 |
| `supports_precise_address()` | PEBS返回true（精确到缓存行），Mock返回false |
| `is_running()` | 查询当前状态 |

### 2.2 `TracerFactory` — 环境自动感知
**文件：** `backend/src/tracer/tracer_factory.cpp`

工厂类负责在运行时决定创建哪种 Tracer，让同一份代码在 VM 和物理机上都能运行。

#### `TracerFactory::is_pebs_available()`
**作用：** 通过向内核发起一次探测性 syscall 判断当前环境是否支持 Intel PEBS 硬件采样。

```cpp
bool TracerFactory::is_pebs_available() {
    struct perf_event_attr pe;
    pe.type      = PERF_TYPE_RAW;
    pe.config    = 0x1cd;    // MEM_LOAD_RETIRED.L3_MISS 的 Intel 事件编码
                             // 0x1cd = 十进制461，专门抓L3 miss事件
    pe.sample_period = 10000; // 每10000次L3 miss采样一次（避免性能开销过大）
    pe.precise_ip = 2;        // PEBS精确模式，ip误差为0（非PEBS只能做到±几条指令）
    pe.disabled  = 1;         // 创建后先不激活

    int fd = syscall(__NR_perf_event_open, &pe, -1, 0, -1, 0);
    // 参数：事件描述，pid=-1(不绑定进程)，cpu=0，group_fd=-1，flags=0
    
    if (fd >= 0) {
        close(fd);   // 只是探测，立即关闭
        return true; // 内核允许打开 → 物理机环境，支持PEBS
    }
    return false;    // ENODEV/EPERM → VM环境，没有PMU硬件
}
```

**为什么用 `0x1cd`？**  
这是 Intel Sandy Bridge 以后所有 Intel CPU 通用的 L3 Miss 事件原始编码。AMD 有不同的编码，ARM 又不同。这也是为什么项目说"需要 Intel CPU"。

#### `TracerFactory::detect_best_tracer()`
**作用：** 按优先级尝试各种 Tracer，返回最佳类型。

```
优先级：PEBS（硬件精度）> Mock（兼容性）
目前 SOFTWARE（eBPF软件采样）尚未实现，直接 fallback 到 Mock
```

#### `TracerFactory::create(TracerType type)`
**作用：** 根据类型实例化具体 Tracer。`AUTO` 类型先调 `detect_best_tracer()` 再决定。关键安全逻辑：即使用户强制指定 `PEBS`，仍会再次检测是否可用，不可用则 fallback Mock，**避免崩溃**。

### 2.3 `MockTracer` — 虚拟机下的事件生成器
**文件：** `backend/src/tracer/mock_tracer.cpp`

在无硬件 PMU 的环境（VMware、WSL、普通云主机）下，通过统计分布**人工合成**内存访问事件。

#### `MockTracer::initialize(pid_t target_pid)`
```cpp
bool MockTracer::initialize(pid_t target_pid) {
    target_pid_ = target_pid;  // 记录目标PID（Mock模式下仅用于打印日志）
    current_index_ = 0;        // Replay模式的读取游标归零
    return true;               // 始终成功（无硬件依赖）
}
```

#### `MockTracer::start()`
设置 `running_=true`，根据 `preloaded_trace_` 是否为空决定进入哪种工作模式，打印模式信息到 stdout。

#### `MockTracer::collect_samples()` ★ 核心函数
每次被 `TimingAnalyzer::process_epoch()` 调用，返回一批事件。

```cpp
std::vector<MemoryAccessEvent> MockTracer::collect_samples() {
    // ── 模式1：Replay（用户预录的真实Trace）──
    if (!preloaded_trace_.empty()) {
        size_t batch_size = std::min(size_t(1000), remaining);
        // 每次取最多1000个事件，维护 current_index_ 游标
        // loop_mode_=true 时，游标到头后归零，无限循环
        return samples;
    }

    // ── 模式2：随机生成 ──
    int num_samples = uniform(100, 500);  // 每Epoch随机生成100~500个事件
    for each sample:
        sample = generate_random_event();
    return samples;
}
```

#### `MockTracer::generate_random_event()` ★ 事件生成细节
```cpp
MemoryAccessEvent MockTracer::generate_random_event() {
    e.timestamp_ns = steady_clock::now();  // 真实的系统时间戳

    // 地址生成：在配置的地址范围内均匀随机，然后 & ~0x3F 对齐到64字节缓存行
    // 原因：CPU访存总是以64字节缓存行为单位，非对齐地址不真实
    e.virtual_addr = uniform(addr_start_, addr_end_) & ~0x3F;

    e.is_load = true;  // 目前只生成 load，store 未来扩展

    // L3 miss 决策：按配置的 miss_rate（默认20%）伯努利采样
    e.is_l3_miss = (uniform(0,1) < l3_miss_rate_);  // 20%概率为true

    if (e.is_l3_miss) {
        // L3 miss：延迟服从正态分布 N(avg_latency_ns, 20% * avg)
        // 150ns平均，std=30ns，模拟真实内存访问的延迟分布
        e.latency_cycles = max(50, normal(avg_latency_ns_, avg_latency_ns_*0.2));
    } else {
        // L3 hit：均匀分布 [10ns, 30ns]，模拟L3缓存命中延迟
        e.latency_cycles = uniform(10, 30);
    }
    return e;
}
```

#### `MockTracer::load_trace_file(json_path)`
从 JSON 文件加载预录的真实 Trace，JSON 格式：
```json
{
  "events": [

## 三、仿真核心层 — 延迟计算

### 3.1 `TopologyGraph` — CXL网络的图模型
**文件：** `backend/src/topology/topology_graph.cpp`

用**有向加权图**表示 CXL 物理互连拓扑。

**内部数据结构：**
```
nodes_: std::map<string, TopologyNode>
    key = 节点ID（如 "RC0", "SW1", "CXL_MEM1"）
    value = TopologyNode { type, union{ rc_data, switch_data, cxl_device_data } }

edges_: std::map<pair<string,string>, TopologyEdge>
    key = (from_id, to_id) 有向边
    value = TopologyEdge { bandwidth_gbps, latency_ns, pcie_generation, 
                           link_width, current_load }
```

#### `TopologyGraph::build_from_config(config)` ★
**作用：** 将用户在 GUI 中配置的拓扑结构转化为图数据结构，是仿真启动前的关键初始化步骤。

执行顺序：
```
1. 清空所有节点和边（支持重建）

2. 添加 Root Complex 节点
   → 从 config.root_complex 读取 DRAM大小、本地DRAM延迟

3. 遍历 config.switches，逐个添加 Switch 节点
   → 记录 processing_latency_ns（交换机转发延迟，通常15~30ns）
   → 记录 num_ports（端口数）

4. 遍历 config.cxl_devices，逐个添加 CXL Device 节点
   → 记录 capacity_gb（设备内存容量）
   → 记录 base_latency_ns（设备自身访问延迟，Gen5约170ns）
   → 记录 supports_coherency（是否支持 CXL.cache 缓存一致性）

5. 遍历 config.connections，添加有向边
   → 解析链路规格字符串（如 "PCIe5.0-x16"）
   → Gen5 x16 = 64 GB/s，Gen4 x16 = 32 GB/s，x8减半
   → base link latency 固定10ns（PCIe物理信号传输延迟）
   → 同时插入反向边（PCIe是全双工，双向都能传数据）

6. validate() → check_connectivity() 用BFS验证所有CXL设备可达
```

#### `TopologyGraph::add_node(node)`
向 `nodes_` map 插入节点。检查 ID 不为空、不重复，违反则返回 false。O(log N) 复杂度。

#### `TopologyGraph::add_edge(edge)`
向 `edges_` map 插入有向边。Key 是 `(from_id, to_id)` pair，同一方向只能有一条边（后插覆盖前插）。O(log E) 复杂度。

#### `TopologyGraph::find_path(from_id, to_id)`
对外暴露的路径查询接口，内部直接调用 `dijkstra()`，返回 `TopologyPath`（包含路径节点列表、总延迟、最小带宽、边指针列表）。

#### `TopologyGraph::dijkstra(from, to)` ★★ 最关键算法
**作用：** 找从 Root Complex 到目标 CXL 设备的**最低延迟路径**。

```
数据结构：
  distances: map<string, double>   // 每个节点到起点的当前最短延迟
  previous:  map<string, string>   // 前驱节点，用于路径重建
  visited:   set<string>           // 已处理节点
  pq: priority_queue<(dist, id), min-heap>  // 按延迟排序的最小堆

算法流程：
  1. 初始化：所有距离=∞，起点距离=0，起点入队
  2. 循环取出堆顶（当前最近未处理节点）：
     a. 若已visited，跳过（可能是过期的副本）
     b. 若到达目标节点，break（最优路径确定）
     c. 遍历所有邻居：
        new_dist = distances[current] + edge.latency_ns
        若邻居是Switch：new_dist += switch.processing_latency_ns
        若 new_dist < distances[neighbor]：更新并入队
  3. 路径重建：从 to 沿 previous 指针反向回溯到 from，reverse

代价函数 = 链路传输延迟 + 交换机处理延迟
（不包含设备自身延迟和拥塞，那些在 LatencyModel 里叠加）
```

**为什么路径是稳定的（不会每次都变）？**  
在拓扑不变的情况下，Dijkstra 每次对相同 (from, to) 返回相同路径。`TopologyGraph` 在仿真期间不重建，因此路径查询结果可以被隐式缓存（实际代码每次都重新算，但结果相同）。

#### `TopologyGraph::get_neighbors(node_id)`
遍历 `edges_` 找所有以 `node_id` 为起点的边，返回终点列表。O(E) 复杂度（未建邻接表，可优化）。

#### `TopologyGraph::check_connectivity()`
用 **BFS（广度优先搜索）** 从 Root Complex 出发，检查所有 CXL_DEVICE 节点是否都可达。如果有设备"孤岛"（没有连线），`build_from_config` 会失败，`onStartSimulation` 会弹出警告。

#### `TopologyGraph::reset_link_loads()`
每个 Epoch 开始时调用，将所有边的 `current_load` 清零，准备重新统计本轮带宽占用。

#### `TopologyGraph::update_link_load(from, to, bandwidth_used)`
更新某条边的负载比例：
```cpp
edges_[key].current_load = bandwidth_used / edges_[key].bandwidth_gbps;
// 钳位到 [0.0, 1.0]，0.0=空闲，1.0=满载
```

#### `TopologyGraph::get_link_load(from, to)`
查询某条边当前负载，供 `LatencyModel::calculate_congestion_penalty()` 使用。

### 3.2 `LatencyModel` — 五因素延迟分解
**文件：** `backend/src/analyzer/latency_model.cpp`

将一次 CXL 内存访问的总延迟拆解为五个物理来源之和，每个都有明确的硬件对应物。

#### `LatencyModel::calculate_latency(device_id, topology)` ★★
**作用：** 计算访问指定设备的**结构性延迟**（不含随机抖动），返回完整的 `LatencyBreakdown`。

```
五因素计算：

① base_latency_ns = device->data.cxl_device.base_latency_ns
   物理含义：CXL DRAM控制器接收到请求后，到DRAM芯片响应的时间
   典型值：CXL Gen5 约 170ns（相比本地DRAM的 80~90ns 慢约2倍）
   来源：用户在GUI拓扑图中配置的CXL设备属性

② link_latency_ns = Σ(edge.latency_ns) for each edge in path
   物理含义：信号在PCIe总线上的传输时延，每条链路约10ns
   RC→Switch：10ns，Switch→Device：10ns，共20ns（两跳）
   来源：build_from_config 中硬编码的 edge.latency_ns=10.0

③ switch_latency_ns = Σ(switch.processing_latency_ns) for each switch in path
   物理含义：CXL Switch的报文处理延迟（路由决策、流控、缓冲）
   典型值：15~30ns，取决于Switch芯片型号
   来源：用户在GUI中配置的Switch节点属性

④ protocol_overhead_ns = base_latency_ns × protocol_overhead_ratio
   物理含义：CXL协议层开销，包括：
   - TLP（事务层报文）封装/解析
   - FLIT（68字节或256字节流控单元）打包
   - CRC校验计算
   典型值：base的5%，约 8~15ns
   计算：calculate_protocol_overhead(base_latency_ns) = base × ratio

⑤ congestion_penalty_ns = calculate_congestion_penalty(path, topology)
   物理含义：链路拥塞时请求在队列中等待的时间
   触发条件：某条链路 current_load > 60%
   计算方式：见下方详解
```

`breakdown.calculate_total()` 最后将五项求和写入 `total_ns`。

#### `LatencyModel::calculate_access_latency(event, device_id, topology)` ★★
**作用：** 在结构性延迟基础上叠加**per-sample随机抖动**，使每次访问延迟有真实的统计分布。

```
step1: 调用 calculate_latency() 得到 base（结构性延迟）

step2: 构造随机种子
   seed = (event.virtual_addr >> 6) ^ event.timestamp_ns
   为什么这样做：
   - virtual_addr >> 6 去掉缓存行内偏移，以缓存行为粒度区分地址
   - XOR timestamp_ns 引入时间维度，同地址不同时刻有不同抖动
   - 确定性种子：相同事件产生相同抖动（可重现）

step3: 高斯抖动（时钟/总线噪声）
   jitter ~ N(0, σ=12% × base)
   σ=12% 来自实测：Intel的PMU测量表明DDR5延迟标准差约10~15%

step4: 写操作惩罚
   write_penalty = is_load ? 0 : base × 10%
   写操作需要等待数据从CPU缓存flush到内存，比读操作慢

step5: 尾延迟事件（5%概率触发）
   spike ~ Exp(λ = 1/(base×0.5))，期望值=base×0.5
   物理含义：偶发的额外排队等待（链路仲裁冲突、RefreshCollide等）
   5%概率 × 平均0.5×base的额外延迟 → P99会比avg高约25~40%

step6: 下限保护
   total = max(total, base × 0.5)  不允许低于base的50%

step7: 可选MLP优化（当前度数=1，实际无效果）
```

#### `LatencyModel::calculate_congestion_penalty(path, topology)` ★
**作用：** 对路径上每条超载链路计算非线性排队延迟惩罚。

```
触发条件：load > 0.6（60%）

对每条超载边：
   queue_factor = (load - 0.6) / (1 - 0.6)  // 归一化到[0,1]
   queue_depth  = 16 × queue_factor          // 16 = PCIe典型队列深度
   queue_delay  = edge.latency_ns × queue_depth × (1 + queue_factor)
   
   非线性原因：
   - (1 + queue_factor) 这个乘法项使延迟随负载非线性增长
   - load=60%：queue_delay ≈ 0
   - load=80%：queue_delay ≈ 10 × 8 × 1.5 = 120ns
   - load=90%：queue_delay ≈ 10 × 12 × 1.75 = 210ns
   
   突发流量惩罚（load > 90%）：
   burst_factor = exp((load - 0.9) × 10)  // 指数爆炸
   queue_delay ×= burst_factor
   - load=95%：burst_factor = e^0.5 ≈ 1.65
   - load=99%：burst_factor = e^0.9 ≈ 2.46

多链路累积：
   return sqrt(total_penalty × edge_count) × 10
   使用平方根避免多跳路径惩罚线性叠加导致过估
```

#### `LatencyModel::apply_mlp_optimization(base, mlp_degree)`
**作用：** 建模内存级并行（Memory Level Parallelism）对延迟的隐藏效果。

```
effective_latency = base / sqrt(mlp_degree) × params_.mlp_factor
下限：base × 0.5（最多隐藏50%延迟）

原理：当CPU同时发出N个不相关内存请求时，内存控制器可以并行处理，
有效延迟接近 base/√N。当前 mlp_degree 硬编码为1（无并行），
未来可通过 LBR 数据检测真实 MLP 度数。
```

#### `LatencyModel::calculate_protocol_overhead(base)`
返回 `base × protocol_overhead_ratio`。ratio 默认5%，可在配置中调整。

#### `LatencyModel::calculate_flit_overhead(data_size, pcie_gen)`
计算 CXL Flit 打包开销：Gen5 使用256字节大 Flit（效率93%），Gen4/3 使用68字节小 Flit（效率90%）。每个 Flit 约1.5ns额外开销。当前主流程未调用，留作扩展。

#### `LatencyModel::calculate_snoop_overhead(requires_snoop)`
CXL.cache 缓存一致性的 Snoop 开销。`requires_snoop=true` 时返回固定40ns（Snoop广播+响应的典型延迟）。

#### `TimingAnalyzer::start()` ★
```cpp

    if (running_) return false;   // 防止重复启动
    if (!tracer_) return false;   // 没有Tracer无法工作

```

#### `TimingAnalyzer::stop()`
```cpp
void TimingAnalyzer::stop() {
    running_ = false;            // atomic<bool>，对后台线程可见
    analyzer_thread_.join();     // 等待后台线程退出循环（最多等一个Epoch=10ms）
    tracer_->stop();             // 关闭硬件计数器/mmap
}
```

#### `TimingAnalyzer::analyzer_loop()` ★★
**作用：** 后台线程的主循环，精确对齐到 Epoch 边界。

```cpp
void TimingAnalyzer::analyzer_loop() {
    auto epoch_duration = chrono::milliseconds(config_.epoch_duration_ms); // 10ms

    while (running_) {
        auto epoch_start = steady_clock::now();

        process_epoch();   // 核心计算，耗时不定（取决于样本数量）
        total_epochs_++;

        // 计算本次处理耗时，用剩余时间补眠，对齐到10ms边界
        auto elapsed = steady_clock::now() - epoch_start;
        auto sleep_time = epoch_duration - elapsed;

        if (sleep_time > 0ms) {
            this_thread::sleep_for(sleep_time);
        }
        // 若处理时间超过10ms，直接进入下一轮（丢失时间对齐但不会堆积）
    }
}
```

**为什么要对齐到Epoch边界？** 带宽计算用的分母是 `epoch_duration_ms / 1000`，如果每轮时长不一致，带宽数值会波动。通过补眠对齐，保证每轮都是稳定的10ms时间窗口。

#### `TimingAnalyzer::process_epoch()` ★★★ 最核心函数
**作用：** 执行一个完整的模拟周期，产出该 Epoch 的全部统计数据。

```
执行步骤（完整版）：

① current_stats_ = EpochStats()
   归零本轮统计，设置 epoch_number = total_epochs_

② samples = tracer_->collect_samples()
   拿到本轮100~500个事件。若为空（Tracer未启动、Trace播完），
   直接return，本Epoch的stats全为0

③ update_bandwidth_stats(samples)
   → reset_link_loads()：清空所有边的 current_load
   → 统计各设备访问次数：device_access_count[device_id]++
   → 计算每设备带宽：count × 64B / epoch_duration_s / 1e9
   → 找RC到设备的路径（调Dijkstra）
   → 对路径上每条边调 update_link_load(bandwidth)
   这一步让链路有了"当前负载"，供后续拥塞惩罚计算用

④ 主处理循环：for each sample
   if sample.is_l3_miss:
     device_id = find_device_for_address(sample.virtual_addr)
     
     if device_id非空（命中CXL地址范围）：
       cxl_accesses++
       latency_ns = latency_model_.calculate_access_latency(sample, device_id, topology_)
       total_latency += latency_ns
       cxl_latencies.push_back(latency_ns)
       
       if config_.enable_injection:
         inject_delay(latency_ns)        ← 真实阻塞当前线程！
         total_injected_delay_ns += latency_ns
     
     else（地址不在CXL范围，本地DRAM）：
       local_dram_accesses++
   
   else（L3 hit，不计入CXL统计）：
     （不做任何处理，只在total_accesses里计数）

⑤ 统计计算
   avg = total_latency / cxl_accesses
   
   sort(cxl_latencies)  ← 排序后取百分位
   p95 = cxl_latencies[size × 0.95]
   p99 = cxl_latencies[size × 0.99]
   
   queuing_delay = avg × 0.1  （简化：拥塞延迟约为均值10%）
   
   link_utilization_pct = cxl_accesses × 64B / epoch_s / 1e9 / 64GB/s × 100
   （64 GB/s = CXL 2.0单通道峰值带宽）

⑥ 每10轮打印一次统计到 stdout（verbose模式）
```

#### `TimingAnalyzer::find_device_for_address(addr)` 
线性扫描 `address_mappings_`，调用每个 `AddressMapping::contains(addr)`，返回第一个匹配的 `device_id`，未找到返回空串。

#### `TimingAnalyzer::update_bandwidth_stats(samples)`
详见 `process_epoch()` 第③步。注意：只处理 `is_l3_miss=true` 的样本，因为L3 hit不经过PCIe总线，不占用CXL链路带宽。

#### `TimingAnalyzer::inject_delay(delay_ns)` ★
**作用：** 通过混合策略在仿真进程中注入真实的延迟，使被仿真的应用程序感受到真实的 CXL 延迟。

```cpp
void TimingAnalyzer::inject_delay(uint64_t delay_ns) {
    // ① 加入±3%随机抖动（inject本身也有误差）
    double jitter = 1.0 + (rand()/RAND_MAX - 0.5) × 0.06;
    uint64_t actual_delay = delay_ns × jitter;

    const uint64_t SPIN_THRESHOLD = 10000;  // 10微秒边界

    // ② 混合策略：
    if (actual_delay > 10000ns) {
        // 大部分时间用 sleep_for（让出CPU，省电）
        sleep_for(actual_delay - 10000ns);
        actual_delay = 10000ns;  // 剩余10微秒用自旋精确控制
    }

    // ③ 高精度自旋：用高精度时钟轮询，精度约100ns
    auto start = high_resolution_clock::now();
    while (elapsed < actual_delay) {
        if (remaining > 100ns) yield();  // 让出CPU避免过度占用
    }
}
```

**为什么不全用 sleep？** `sleep_for` 最小精度约1ms（Linux内核调度粒度），对于150~300ns的延迟注入，全用sleep会差1000倍。**为什么不全用自旋？** 纯自旋会吃满一个CPU核，多个设备访问并发时CPU被耗尽，影响被仿真应用的性能。混合策略：大延迟sleep省CPU，最后10μs自旋保精度。

#### `TimingAnalyzer::run_one_epoch()`
手动触发一个 Epoch 并返回统计结果，专用于**单元测试**，生产运行中不调用。

## 六、核心设计决策总结

| 设计点 | 选择 | 原因 |
|--------|------|------|
| 采集与分析解耦 | `ITracer` 抽象接口 | VM/物理机零代码改动切换 |
| 后台独立线程 | `std::thread` | 不阻塞 GUI，Epoch 定时精确 |
| 跨线程通信 | Qt 信号槽 QueuedConnection | 无需手动加锁，线程安全 |
| 延迟注入精度 | sleep + 自旋混合 | 平衡精度（<100ns误差）与CPU开销 |
| 路径查找 | Dijkstra | 支持任意拓扑（树形/Mesh），O((V+E)logV) |
| P95/P99 计算 | 排序取百分位 | 简单正确，每Epoch最多500个样本，O(N logN) 可接受 |
| 随机抖动种子 | addr XOR timestamp | 确定性可重现 + 每事件不同，不使用全局random避免线程竞争 |
```
